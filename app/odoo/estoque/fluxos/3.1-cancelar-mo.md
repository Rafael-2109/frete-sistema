# Fluxo 3.1 — Cancelar Manufacturing Order

**Quando:** cancelar 1 ou mais `mrp.production` SEM gerar furo contábil (componentes consumidos sem produto finalizado). Três sub-casos:
- **3.1.a** Cancelar 1 MO específica (zumbi reportada, state IN draft/confirmed/progress/to_close, consumo = 0).
- **3.1.b** Cancelar batch de MOs antigas (zumbis) por critério (data, states, empresas, consumo zero).
- **3.1.c** MO COM consumo > 0 → **NÃO coberto** por esta skill. Delegar para fluxo `mrp.unbuild` (sem skill ainda — ver memória `[[reaproveitar-semiacabado-orfao-mo-cancelada]]`).

**Galho da árvore (subagente):** `3 Produção/PCP → 3.1 cancelar MO`.

---

## Skill atômica

`operando-mo-odoo` — service em `app/odoo/estoque/scripts/mo.py` (StockMOService). CLI em `.claude/skills/operando-mo-odoo/scripts/operar_mo.py`.

2 átomos:
- `svc.cancelar_mo(mo_id, motivo, forcar_consumo, consumo_total, dry_run)` — wrapper sobre `mrp.production.action_cancel` + G019-like re-le state pós; guard G-MO-01 (consumo > 0 = FALHA_FURO_CONTABIL) por default.
- `svc.cancelar_mos_em_massa(criterio, max_n, motivo, dry_run)` — composição. Filtra MOs por (create_de, create_ate, states, empresas, consumo), mede consumo em batch (perf), delega `cancelar_mo` por MO.

Helper:
- `svc.medir_consumo_mo(mo_ids)` — soma `stock.move.quantity` (state != 'cancel') por `raw_material_production_id`. Tolerância > 0.0001. Usado por guard G-MO-01 e por filtro batch.

## Premissas (pesquisar+validar ANTES de qualquer write)

### Comum aos 3 sub-casos
1. **MO existe** (`mrp.production.read` com `name`, `state`, `company_id`).
2. **Estado da MO** alinhado com modo:
   - `cancelar` exige `state IN (draft, confirmed, progress, to_close)`.
   - Se `state='cancel'` → NOOP (idempotente).
   - Se `state='done'` → FALHA_STATE_NAO_CANCELAVEL (não tem como reverter sem unbuild).
3. **Guard G-MO-01** (sempre): consumo_total > 0 = FALHA_FURO_CONTABIL. Default seguro.
4. **Implicação operacional** (CRÍTICO se MO em `progress`):
   - Operador no chão de fábrica pode estar apontando ATIVAMENTE. Cancelar perde o apontamento.
   - Default `--states draft,confirmed,progress,to_close` INCLUI progress (consistência com scripts-fonte) mas recomenda-se `confirmed,draft` em batches grandes (zumbis raramente estão em progress real).

### Específico 3.1.a (cancelar 1 MO)
5. **Identificar MO** via Skill 9 ou investigação manual (`mrp.production` query por name/origin/operador).
6. **Motivo registrado** para auditoria (string livre, não afeta Odoo — só log).

### Específico 3.1.b (batch zumbis)
7. **Critério restritivo** (recomendado): `create_ate` (excluir MOs recentes) + `empresas` (FB típica; CD quase inativo; LF poucas zumbis) + `consumo zero` (default).
8. **Canary obrigatório**: `--limite 1` antes de batch completo. Verificar resultado direto no Odoo.
9. **MOs de SEMI-ACABADO multi-nível**: cancelar MO de acabado pode deixar MO de semi órfã (caso real `[[reaproveitar-semiacabado-orfao-mo-cancelada]]`). Esta skill NÃO trata isso — operador valida manualmente após cancel.

### Específico 3.1.c (MO COM consumo > 0 — DELEGADO)
10. **NÃO COBERTO por esta skill**. Procedimento via `mrp.unbuild` documentado em [[reaproveitar-semiacabado-orfao-mo-cancelada]] §3:
    1. Criar `mrp.unbuild` com `mo_id` definido (devolve componentes aos lotes originais automaticamente).
    2. `action_unbuild` (retorna None via XML-RPC = sucesso).
    3. Validar saldos restaurados nos lotes originais.

## Composição (3 sub-casos)

### 3.1.a — Cancelar 1 MO (zumbi reportada)

```bash
SK=.claude/skills/operando-mo-odoo/scripts/operar_mo.py

# 1) Dry-run: ver state, consumo, plano
python "$SK" --modo cancelar --mo-id 19713

# 2) Se OK (state cancelável + consumo=0), efetivar
python "$SK" --modo cancelar --mo-id 19713 --motivo "zumbi 2025" --confirmar

# 3) (Cruzar) Se quants de componentes tinham reserva residual após cancel
#    (raro — action_cancel libera tudo em cascade) → cleanup
python .claude/skills/operando-reservas-odoo/scripts/operar_reserva.py \
    --zerar-residual --quant-ids QUANT_IDS --confirmar
```

### 3.1.b — Batch zumbis antigas (pattern 2026-05-20, 120 MOs)

```bash
# 1) Dry-run amplo (entende o universo)
python "$SK" --modo cancelar --create-ate 2025-06-01 --empresas 1 --consumo zero

# 2) Revisar lista no log JSON salvo automaticamente
ls -lh scripts/inventario_2026_05/auditoria/log_skill4_mo_dryrun_*.json | tail -1

# 3) Canary: 1 MO real (NUNCA pular)
python "$SK" --modo cancelar --create-ate 2025-06-01 --empresas 1 \
    --consumo zero --limite 1 --confirmar
# → Verificar diretamente no Odoo: MO state='cancel', moves cancelados.

# 4) Batch completo (cuidado!)
python "$SK" --modo cancelar --create-ate 2025-06-01 --empresas 1 \
    --consumo zero --motivo "limpeza zumbi 2024-2025" --confirmar
```

### 3.1.c — MO COM consumo (NÃO COBERTO — DELEGAR)

Procedimento manual via XML-RPC (ver `[[reaproveitar-semiacabado-orfao-mo-cancelada]]` §3):

```python
# 1. mrp.unbuild — devolve componentes aos lotes originais
unbuild_id = odoo.create('mrp.unbuild', {
    'product_id': mo_product_id,
    'mo_id': mo_id,
    'bom_id': mo_bom_id,
    'product_qty': mo_product_qty,
    'product_uom_id': mo_product_uom_id,
    'lot_id': mo_lot_id_finished,  # lote do produto acabado
    'company_id': mo_company_id,
    'location_id': mo_location_dest_id,  # onde produto acabado esta
    'location_dest_id': mo_location_src_id,  # devolver para origem
})
odoo.execute_kw('mrp.unbuild', 'action_unbuild', [[unbuild_id]])
# 2. Validar saldos via Skill 9 (consultando-quant-odoo)
```

## Pós-execução (verificação obrigatória)

1. **3.1.a (1 MO)**: `svc.cancelar_mo` retorna `EXECUTADO` com `state_apos='cancel'`. Verificar:
   - `mrp.production.state='cancel'` (`stock.move` filhas também viraram cancel por cascade).
   - Reservas de componentes liberadas (`stock.quant.reserved_quantity` recalculado).
   - Se sobrar reserved residual stale → Skill 2.4 `zerar_reserved_residual`.

2. **3.1.b (batch)**: `svc.cancelar_mos_em_massa` retorna dict com `contagem_status` agregada:
   - `EXECUTADO`: N MOs canceladas com sucesso.
   - `NOOP`: M MOs já estavam em cancel (idempotente).
   - `FALHA_FURO_CONTABIL`: P MOs bloqueadas (consumo > 0 — preservadas).
   - `FALHA_STATE_INESPERADO`: Q MOs onde action_cancel não mudou state (investigar caso a caso).
   - Verificar via Skill 9 que critério agora retorna 0 candidatas (idempotência batch).

3. **3.1.c (DELEGADO)**: validar via Skill 9 que componentes voltaram aos lotes originais com saldo esperado.

## Gotchas-invariante (codificados no service)

- **G-MO-01 (FURO CONTÁBIL)**: consumo > 0 default bloqueado (`FALHA_FURO_CONTABIL`). Parâmetro `forcar_consumo=True` existe no service mas **CLI V1 NÃO expõe** — operador deve usar fluxo `mrp.unbuild` cross-skill.
- **G-MO-02 (manual_consumption ≠ action_assign)**: Não relevante para cancelar (action_cancel ignora reservas/picked). Relevante para criar/alterar MO (não cobertos em V1).
- **G-MO-03 (componente em local errado)**: Não relevante para cancelar. Relevante se for alterar MO (mover componente entre locais antes de assignar — ver `[[mo_componente_local_consumo]]`).
- **G-MO-04 (picked=True em to_close/done)**: Herdado de Skill 2.4 G026. `action_cancel` é seguro com picked (não mexe em quants existentes — só marca state='cancel').
- **G019-like**: SEMPRE re-le state pós `action_cancel`. Em casos extremos (MO referenciada por outra em produção, lock pessimista, regra customizada), action_cancel pode falhar silenciosamente → service retorna `FALHA_STATE_INESPERADO`.
- **Idempotência**: state pré='cancel' → NOOP sem chamar Odoo novamente (economiza RPC). Validado AO VIVO 2026-05-24 (FB/OP/BALDE/00009 id=4192 retornou `True` sem erro, state continuou 'cancel').
- **`stock.move.action_cancel` NÃO existe** via XML-RPC (`_action_cancel` é privado). Cancelar MO via `mrp.production.action_cancel` é o caminho correto (cascade automático para moves + MLs).
- **`qty_produced` ≠ consumo**: `qty_produced` é o produto acabado finalizado. Para guard G-MO-01, medimos `stock.move.quantity` (raw materials != cancel), não `qty_produced`.
- **TOL_CONSUMO=0.0001** (mesma dos scripts-fonte). Consumos abaixo são tratados como zero (rounding errors do Odoo: 6 decimais).
- **MOs zumbi multi-nível**: cancelar MO de acabado pode deixar MO de semi órfã (caso real [[reaproveitar-semiacabado-orfao-mo-cancelada]]). Esta skill NÃO trata cross-nível — operador valida manualmente.
