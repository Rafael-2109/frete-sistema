# Fluxo 3.1 — Cancelar Manufacturing Order

**Quando:** cancelar 1 ou mais `mrp.production` SEM gerar furo contábil REAL (componentes com baixa contábil efetivada `state='done'`). Três sub-casos:
- **3.1.a** Cancelar 1 MO específica (zumbi reportada, state IN draft/confirmed/progress/to_close, classificacao SEGURO ou RESERVA_FANTASMA).
- **3.1.b** Cancelar batch de MOs antigas (zumbis) por critério (data, states, empresas, consumo zero — agora filtra apenas FURO_REAL).
- **3.1.c** MO com `done > 0` (FURO_REAL) → **NÃO coberto** por esta skill. Delegar para fluxo `mrp.unbuild` (sem skill ainda — ver memória local Claude Code `[[reaproveitar-semiacabado-orfao-mo-cancelada]]`).

**Galho da árvore (subagente):** `3 Produção/PCP → 3.1 cancelar MO`.

---

## Skill atômica

`operando-mo-odoo` v6 (2026-05-27) — service em `app/odoo/estoque/scripts/mo.py` (StockMOService). CLI em `.claude/skills/operando-mo-odoo/scripts/operar_mo.py`. Skill mista WRITE+READ (ver constituição §6.b).

**5 átomos** (3 WRITE + 2 READ):

WRITE:
- `svc.cancelar_mo(mo_id, motivo, forcar_consumo, consumo_total, dry_run)` — wrapper sobre `mrp.production.action_cancel` + G019-like re-le state pós; guard G-MO-01 v6 (done > 0 = FALHA_FURO_CONTABIL_REAL) por default.
- `svc.cancelar_mo_com_audit(mo_id, ...)` — wrapper sobre `cancelar_mo` capturando snapshot pré/pós + diff estruturado (opt-in custoso).
- `svc.cancelar_mos_em_massa(criterio, max_n, motivo, dry_run)` — composição. Filtra MOs por (create_de, create_ate, states, empresas, consumo), mede consumo em batch (perf), delega `cancelar_mo` por MO.

READ (§6.b — modos READ em skills WRITE):
- `svc.listar_mos(criterio)` — lista MOs candidatas com classificação SEGURO/RESERVA_FANTASMA/FURO_REAL (cada item ganha rótulo conforme consumo `done`/`reservado`).
- `svc.detalhar_mo(mo_id)` — detalhamento completo de 1 MO (raws + finished + MLs + consumo + quants origem).

Helpers:
- `svc.medir_consumo_mo(mo_ids)` v6 — retorna `{mo_id: {done, reservado, total}}` particionando `stock.move.quantity` por `state` (`done` vs `assigned|waiting|partially_available|confirmed`). Tolerância > 0.0001. Usado por guard G-MO-01 v6, filtro batch e classificação READ.
- `svc.medir_consumo_mo_legacy(mo_ids)` — compat float (`{mo_id: total}`) para callers antigos. Não usar em código novo.
- `svc._snapshot_mo(mo_id)` + `svc._diff_snapshots(pre, pos)` — helpers internos do audit.

## Premissas (pesquisar+validar ANTES de qualquer write)

### Comum aos 3 sub-casos
1. **MO existe** (`mrp.production.read` com `name`, `state`, `company_id`).
2. **Estado da MO** alinhado com modo:
   - `cancelar` exige `state IN (draft, confirmed, progress, to_close)`.
   - Se `state='cancel'` → NOOP (idempotente).
   - Se `state='done'` → FALHA_STATE_NAO_CANCELAVEL (não tem como reverter sem unbuild).
3. **Guard G-MO-01 v6 (refinado 2026-05-27)** — particiona consumo:
   - `done > TOL` (consumo CONTÁBIL `state='done'`) → FALHA_FURO_CONTABIL_REAL. Default seguro INVIOLÁVEL.
   - `reservado > TOL` e `done = 0` (apenas reserva fantasma — picked=True em assigned/waiting) → PASSA com `OK_RESERVA_FANTASMA` (action_cancel libera reservas sem furo).
   - `done = 0` e `reservado = 0` → `OK` (limpo).
   - Origem da v6: caso real 2026-05-27 — 29 MOs zumbi pré-2026-05-15 classificadas como FURO em v5 conservador; auditoria revelou 100% FALSO-POSITIVO (todos os moves em `assigned`/`waiting`, nenhum em `done`). Ver `docs/inventario-2026-05/02-gotchas/G-MO-05-falso-positivo-reserva-fantasma.md`.
4. **Investigação prévia (recomendada — modos READ §6.b)**: usar `--modo listar` para enxergar candidatas+classificação e `--modo detalhar` para inspecionar MO suspeita antes de qualquer WRITE.
5. **Implicação operacional** (CRÍTICO se MO em `progress`):
   - Operador no chão de fábrica pode estar apontando ATIVAMENTE. Cancelar perde o apontamento.
   - Default `--states draft,confirmed,progress,to_close` INCLUI progress (consistência com scripts-fonte) mas recomenda-se `confirmed,draft` em batches grandes (zumbis raramente estão em progress real).

### Específico 3.1.a (cancelar 1 MO)
6. **Identificar MO** via Skill 9 (`consultando-quant-odoo`) ou modo `listar` da própria skill (`--create-ate X --empresas Y --limite N`).
7. **Auditar 1 MO**: `--modo detalhar --mo-id <id>` → ver raws+finished+MLs+consumo+classificacao.
8. **Motivo registrado** para auditoria (string livre, não afeta Odoo — só log).

### Específico 3.1.b (batch zumbis)
9. **Critério restritivo** (recomendado): `create_ate` (excluir MOs recentes) + `empresas` (FB típica; CD quase inativo; LF poucas zumbis) + `consumo zero` (default — v6 filtra apenas FURO_REAL; FANTASMA passa).
10. **Canary obrigatório**: `--limite 1` antes de batch completo. Verificar resultado direto no Odoo.
11. **MOs de SEMI-ACABADO multi-nível**: cancelar MO de acabado pode deixar MO de semi órfã (caso real memória local Claude Code `[[reaproveitar-semiacabado-orfao-mo-cancelada]]`). Esta skill NÃO trata isso — operador valida manualmente após cancel.

### Específico 3.1.c (MO COM `done > 0` — DELEGADO)
12. **NÃO COBERTO por esta skill** (v6 retorna `FALHA_FURO_CONTABIL_REAL`). Procedimento via `mrp.unbuild` documentado em memória local Claude Code `[[reaproveitar-semiacabado-orfao-mo-cancelada]]` §3:
    1. Criar `mrp.unbuild` com `mo_id` definido (devolve componentes aos lotes originais automaticamente).
    2. `action_unbuild` (retorna None via XML-RPC = sucesso).
    3. Validar saldos restaurados nos lotes originais.

## Pós-execução (verificação obrigatória)

1. **3.1.a (1 MO)**: `svc.cancelar_mo` retorna `EXECUTADO` com `state_apos='cancel'`. Verificar:
   - `mrp.production.state='cancel'` (`stock.move` filhas também viraram cancel por cascade).
   - Reservas de componentes liberadas (`stock.quant.reserved_quantity` recalculado).
   - Se sobrar reserved residual stale → Skill 2.4 `zerar_reserved_residual`.

2. **3.1.b (batch)**: `svc.cancelar_mos_em_massa` retorna dict com `contagem_status` agregada:
   - `EXECUTADO`: N MOs canceladas com sucesso (consumo limpo).
   - `OK_RESERVA_FANTASMA`: N' MOs canceladas com aviso de reserva fantasma (v6 — sem furo, libera reservas).
   - `NOOP`: M MOs já estavam em cancel (idempotente).
   - `FALHA_FURO_CONTABIL_REAL`: P MOs bloqueadas (done > 0 — preservadas; exigem mrp.unbuild).
   - `FALHA_STATE_INESPERADO`: Q MOs onde action_cancel não mudou state (investigar caso a caso).
   - Verificar via `--modo listar` que critério agora retorna 0 candidatas (idempotência batch).

3. **3.1.c (DELEGADO)**: validar via Skill 9 que componentes voltaram aos lotes originais com saldo esperado.

## Gotchas-invariante (codificados no service)

- **G-MO-01 v6 (FURO CONTÁBIL REAL vs RESERVA FANTASMA)**: `medir_consumo_mo` particiona consumo por `state`. Guard bloqueia apenas `done > TOL` (`FALHA_FURO_CONTABIL_REAL`). `reservado > TOL` e `done = 0` passa como `OK_RESERVA_FANTASMA` (action_cancel libera reservas sem furo). Refinado em 2026-05-27 v6 — caso 29 MOs falso-positivo. Detalhe: `docs/inventario-2026-05/02-gotchas/G-MO-05-falso-positivo-reserva-fantasma.md`. Parâmetro `forcar_consumo=True` existe no service mas **CLI NÃO expõe** — para MO com `done > 0` real, usar `mrp.unbuild` cross-skill.
- **G-MO-02 (manual_consumption ≠ action_assign)**: Não relevante para cancelar (action_cancel ignora reservas/picked). Relevante para criar/alterar MO (não cobertos em V1).
- **G-MO-03 (componente em local errado)**: Não relevante para cancelar. Relevante se for alterar MO (mover componente entre locais antes de assignar — ver memória local Claude Code `[[mo_componente_local_consumo]]`).
- **G-MO-04 (picked=True em to_close/done)**: Herdado de Skill 2.4 G026. `action_cancel` é seguro com picked (não mexe em quants existentes — só marca state='cancel').
- **G-MO-05 v6 (FALSO-POSITIVO RESERVA FANTASMA)**: `medir_consumo_mo` v5 antigo somava `quantity` indistinto por state → 29 MOs classificadas como FURO em 2026-05-27 eram 100% FALSO-POSITIVO (todos os moves em `assigned`/`waiting`, nenhum em `done`). v6 particiona done vs reservado. Doc: `docs/inventario-2026-05/02-gotchas/G-MO-05-falso-positivo-reserva-fantasma.md`.
- **G019-like**: SEMPRE re-le state pós `action_cancel`. Em casos extremos (MO referenciada por outra em produção, lock pessimista, regra customizada), action_cancel pode falhar silenciosamente → service retorna `FALHA_STATE_INESPERADO`.
- **Idempotência**: state pré='cancel' → NOOP sem chamar Odoo novamente (economiza RPC). Validado AO VIVO 2026-05-24 (FB/OP/BALDE/00009 id=4192 retornou `True` sem erro, state continuou 'cancel').
- **`stock.move.action_cancel` NÃO existe** via XML-RPC (`_action_cancel` é privado). Cancelar MO via `mrp.production.action_cancel` é o caminho correto (cascade automático para moves + MLs).
- **`qty_produced` ≠ consumo**: `qty_produced` é o produto acabado finalizado. Para guard G-MO-01, medimos `stock.move.quantity` particionado (raw materials, state != cancel), não `qty_produced`.
- **TOL_CONSUMO=0.0001** (mesma dos scripts-fonte). Consumos abaixo são tratados como zero (rounding errors do Odoo: 6 decimais).
- **MOs zumbi multi-nível**: cancelar MO de acabado pode deixar MO de semi órfã (caso real memória local Claude Code `[[reaproveitar-semiacabado-orfao-mo-cancelada]]`). Esta skill NÃO trata cross-nível — operador valida manualmente.
- **Modos READ (§6.b)**: `listar`/`detalhar` são READ-only. CLI bloqueia `--confirmar` nesses modos (exit 2). Service `listar_mos`/`detalhar_mo` não recebem `dry_run` (sempre READ).
- **Audit pré/pós (`--with-audit`)**: default ON em single (`--mo-id`), OFF em batch. Captura snapshot completo de MO+raws+finished+MLs+quants origem antes e depois, com diff estruturado. Use para single mode ou batches pequenos onde rastreabilidade é mais importante que throughput.
