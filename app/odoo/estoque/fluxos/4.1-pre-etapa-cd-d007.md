# Fluxo 4.1 — PRE-ETAPA inventario CD/FB D007 (planejar / propor / listar / aprovar)

**Quando:** usuario pede "planeja a pre-etapa CD", "propor pre-etapa para o ciclo X", "lista a Onda 5", "aprova a Onda 5 com hash Y", "qual transferencia interna substitui a NF FB->CD?", "gera plano D007", "regerar plano apos planilha nova", "qual valor da Onda 5/6?".

**Decisao D007** (`docs/inventario-2026-05/00-decisoes/D007-pre-etapa-cd-fb-minimizar-nf.md`): substituir NFs inter-filial CD↔FB (~R$ 32,9 mi) + INDISPONIBILIZAR_LOTE/LOCAL (~R$ 60,5 mi) por **transferencias INTERNAS** dentro da company (mesma fronteira fiscal) + **residual minimo** FB→CD via NF (CFOP 5152) + **ajuste positivo puro** (sem origem) quando ninguem doa. Onda 5 (CD) executa ANTES da Onda 2. Onda 6 (FB) futura.

---

## Premissas (pesquisar + validar ANTES de qualquer write)

1. **Scripts 01 + 02 ja rodaram** na worktree atual:
   - `01_extrair_estoque_odoo.py` → `/tmp/estoque_odoo_2026_05.json` (quants por company)
   - `02_carregar_inventario_xlsx.py` → `/tmp/inventario_fisico_2026_05.json` (inv por company)
   - Sem esses inputs: planejar aborta com FALHA_INPUT_AUSENTE (exit 1).
2. **Ciclo identificado** (default INVENTARIO_2026_05; passar `--ciclo` se diferente).
3. **company_id correto** (4=CD Onda 5 / 1=FB Onda 6).
4. **D007 restricao temporal FB pos-etapa:** se FB ja passou pela propria pre-etapa antes do CD, o caller deve pre-filtrar `inputs['companies']['1']['quants']` para conter apenas quants com `lot_id.name='MIGRAÇÃO'` (so MIGRAÇÃO FB pode doar). Antes da pre-etapa FB, qualquer lote FB pode doar (default).
5. **Banco local com tabela `ajuste_estoque_inventario`** para modos propor/listar/aprovar (PostgreSQL com schema migrado).
6. **Aprovacao hash sha256 anti-replay:** rodar `listar-onda` SEMPRE antes de `aprovar-onda` para pegar o hash fresh.

## Gotchas do fluxo (referencia)

- **G-PRE-01 D007 restricao temporal FB:** APOS pre-etapa FB, so MIGRAÇÃO FB pode doar para CD. Skill NAO infere — caller decide filtrando `inputs['companies']['1']['quants']` antes de chamar planejar. Se passar quants_fb sem filtro pos-FB, plano podera puxar qty de lotes "errados" e quebrar corretude FB.
- **G-PRE-02 FIFO determinístico por quant_id:** o algoritmo ordena doadores por `quant_id` (estavel/reproducivel). Se Odoo recriar quants entre planejamentos, FIFO pode escolher quant diferente — refazer planejar+propor garante.
- **G-PRE-03 Lote inv sem nome → 'P-15/05':** agregacao por LOTE_DEFAULT_SEM_NOME. Se operador escrever "P-15/05" explicitamente em outra linha do inv, sera somado ao agregado (cuidado em planilhas).
- **G-PRE-04 MIGRAÇÃO consolidador NEG:** se MIGRAÇÃO nao esta no inv, eh consolidador (recebe sobras). Se MIGRAÇÃO esta no inv, eh lote alvo (pode receber POS). Comportamentos diferentes — escolha do operador no inventario fisico.
- **G-PRE-05 Outliers cod[0] != 1-4 skipados:** produtos com cod estranho (X-prefix, alfanumerico) sao excluidos. Verificar `outliers_skipados` para garantir que nenhum produto critico foi perdido.
- **G-PRE-06 Custo medio ponderado D004:** usa apenas quants com `qty > 0 AND value != 0`. Se TODOS os quants tem value=0, `custo_medio=0` — positivo puro com custo zero eh aceito mas pode quebrar relatorio financeiro.
- **G-PRE-07 Hash sha256 muda se mudar QUALQUER campo:** id, cod, cid, lote_odoo, qtd_ajuste, acao. **Aprovar com hash de listar-onda anterior bloqueia se ajustes foram alterados** (anti-replay). Workflow: listar → revisar → aprovar IMEDIATAMENTE.
- **G-PRE-08 Idempotencia do propor:** DELETE+INSERT zera+reinit. Re-rodar gera novos IDs → hash muda → re-listar antes de aprovar.
- **G-PRE-09 Onda 2 depende de Onda 5:** Onda 2 (TRANSFERIR_FB_CD residual) pressupoe que os lotes alvo do CD ja foram criados pela Onda 5. Executar 09b_executar_pre_etapa ANTES de scripts da Onda 2.
- **G-PRE-10 Executor 09b NAO eh atomo da Skill 6:** eh orchestrator C3 macro que compoe Skills 1 (PURO) + 2 (POS+NEG). Sem capinagem ainda — operacao viva via script-fonte.

python scripts/inventario_2026_05/09b_executar_pre_etapa.py \
   --company-id 4 --confirmar --usuario rafael
