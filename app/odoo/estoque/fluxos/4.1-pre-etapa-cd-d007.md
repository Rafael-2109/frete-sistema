# Fluxo 4.1 — PRE-ETAPA inventario CD/FB D007 (planejar / propor / listar / aprovar / **executar-onda v9**)

**Quando:** usuario pede "planeja a pre-etapa CD", "propor pre-etapa para o ciclo X", "lista a Onda 5", "aprova a Onda 5 com hash Y", "**executa a Onda 5 do CD**", "**executar 10 produtos da Onda 5**", "**executa o cod X da Onda 5**", "qual transferencia interna substitui a NF FB->CD?", "gera plano D007", "regerar plano apos planilha nova", "qual valor da Onda 5/6?".

**Decisao D007** (`docs/inventario-2026-05/00-decisoes/D007-pre-etapa-cd-fb-minimizar-nf.md`): substituir NFs inter-filial CD↔FB (~R$ 32,9 mi) + INDISPONIBILIZAR_LOTE/LOCAL (~R$ 60,5 mi) por **transferencias INTERNAS** dentro da company (mesma fronteira fiscal) + **residual minimo** FB→CD via NF (CFOP 5152) + **ajuste positivo puro** (sem origem) quando ninguem doa. Onda 5 (CD) executa ANTES da Onda 2. Onda 6 (FB) futura.

---

## Premissas (pesquisar + validar ANTES de qualquer write)

1. **Scripts 01 + 02 ja rodaram** na worktree atual:
   - `01_extrair_estoque_odoo.py` → `/tmp/estoque_odoo_2026_05.json` (quants por company)
   - `02_carregar_inventario_xlsx.py` → `/tmp/inventario_fisico_2026_05.json` (inv por company)
   - Sem esses inputs: planejar aborta com FALHA_INPUT_AUSENTE (exit 1).
2. **Ciclo identificado** (default INVENTARIO_2026_05; passar `--ciclo` se diferente).
3. **company_id correto** (4=CD Onda 5 / 1=FB Onda 6).
4. **D007 restricao temporal FB pos-etapa:** se FB ja passou pela propria pre-etapa antes do CD, o caller deve pre-filtrar `inputs['companies']['1']['quants']` para conter apenas quants com `lot_id.name='MIGRAÇÃO'` (so MIGRAÇÃO FB pode doar). Antes da pre-etapa FB, qualquer lote FB pode doar (default).
5. **Banco local com tabela `ajuste_estoque_inventario`** para modos propor/listar/aprovar (PostgreSQL com schema migrado).
6. **Aprovacao hash sha256 anti-replay:** rodar `listar-onda` SEMPRE antes de `aprovar-onda` para pegar o hash fresh.

## Sequencia (atomos componiveis — Skill 6 expoe **5 modos v9**)

```
1) SK="planejar_pre_etapa.py"

# A. PLANEJAR (READ Odoo + grava JSON+Excel)
python "$SK" --modo planejar --company-id 4 --dry-run            # preview
python "$SK" --modo planejar --company-id 4 --confirmar          # grava /tmp/plano_pre_etapa_4.json + docs/.../plano-pre-etapa-4.xlsx
   → output: {modo: planejar, status: PLANEJADO, produtos_processados: N,
              total_pos, total_neg, total_residual_fb_cd, total_positivos_puros,
              total_warnings, outliers_skipados, output_json_path, output_xlsx_path}

# B. REVISAR Excel (4 abas: Internas / Residual FB-CD / Positivos Puros / Warnings)
#    abrir docs/inventario-2026-05/07-relatorios/plano-pre-etapa-4.xlsx
#    decisao humana: aceita ou ajusta planilha + reroda planejar?

# C. PROPOR (WRITE banco local — DELETE+INSERT em ajuste_estoque_inventario PROPOSTO)
python "$SK" --modo propor --company-id 4 --usuario X --dry-run  # preview
python "$SK" --modo propor --company-id 4 --usuario X --confirmar  # efetiva
python "$SK" --modo propor --company-id 4 --usuario X --backup-pg-dump --confirmar  # cinto+suspensorio
   → output: {modo: propor, status: PROPOSTO, n_antes, n_apos, n_deletados,
              contador: {AJUSTE_CD_TRANSF_INTERNA_POS, _NEG, _POSITIVO_PURO,
                         TRANSFERIR_FB_CD}, total_inserts, backup_path}

# D. LISTAR ONDA (READ banco local + hash sha256)
python "$SK" --modo listar-onda --company-id 4
   → output: {modo: listar-onda, status: LISTADO, total, hash, por_acao,
              valor_total} — hash usado em aprovar-onda

# E. APROVAR ONDA (WRITE banco local com hash check anti-replay)
python "$SK" --modo aprovar-onda --company-id 4 --hash <h> --usuario X --dry-run   # valida hash
python "$SK" --modo aprovar-onda --company-id 4 --hash <h> --usuario X --confirmar # UPDATE para APROVADO
   → output: {modo: aprovar-onda, status: APROVADO, ajustes_aprovados,
              hash_atual, hash_esperado, ts_aprovacao}

# F. EXECUTAR ONDA v9 (DESTA SKILL — composicao C3 macro de Skills 1+2 via orchestrator)
python "$SK" --modo executar-onda --company-id 4 --dry-run                                     # preview composicao
python "$SK" --modo executar-onda --company-id 4 --limite 1 --usuario X --confirmar            # canary 1 produto
python "$SK" --modo executar-onda --company-id 4 --limite 10 --usuario X --confirmar           # sub-piloto 10
python "$SK" --modo executar-onda --company-id 4 --max-workers 5 --usuario X --confirmar       # bulk paralelo
python "$SK" --modo executar-onda --company-id 4 --cod-produto X --usuario X --confirmar       # 1 produto especifico (retomada)
   → output: {modo: executar-onda, status: EXECUTADO_ONDA, ajustes_total,
              produtos_total, contadores: {produtos_ok, produtos_parcial,
              produtos_falha, pos_ok, pos_falha, neg_ok, neg_falha,
              puro_ok, puro_falha}, produtos: [{cod, product_id, sucessos,
              falhas, pos/neg/puro_results}], tempo_ms}
```

## Gotchas do fluxo (referencia)

- **G-PRE-01 D007 restricao temporal FB:** APOS pre-etapa FB, so MIGRAÇÃO FB pode doar para CD. Skill NAO infere — caller decide filtrando `inputs['companies']['1']['quants']` antes de chamar planejar. Se passar quants_fb sem filtro pos-FB, plano podera puxar qty de lotes "errados" e quebrar corretude FB.
- **G-PRE-02 FIFO determinístico por quant_id:** o algoritmo ordena doadores por `quant_id` (estavel/reproducivel). Se Odoo recriar quants entre planejamentos, FIFO pode escolher quant diferente — refazer planejar+propor garante.
- **G-PRE-03 Lote inv sem nome → 'P-15/05':** agregacao por LOTE_DEFAULT_SEM_NOME. Se operador escrever "P-15/05" explicitamente em outra linha do inv, sera somado ao agregado (cuidado em planilhas).
- **G-PRE-04 MIGRAÇÃO consolidador NEG:** se MIGRAÇÃO nao esta no inv, eh consolidador (recebe sobras). Se MIGRAÇÃO esta no inv, eh lote alvo (pode receber POS). Comportamentos diferentes — escolha do operador no inventario fisico.
- **G-PRE-05 Outliers cod[0] != 1-4 skipados:** produtos com cod estranho (X-prefix, alfanumerico) sao excluidos. Verificar `outliers_skipados` para garantir que nenhum produto critico foi perdido.
- **G-PRE-06 Custo medio ponderado D004:** usa apenas quants com `qty > 0 AND value != 0`. Se TODOS os quants tem value=0, `custo_medio=0` — positivo puro com custo zero eh aceito mas pode quebrar relatorio financeiro.
- **G-PRE-07 Hash sha256 muda se mudar QUALQUER campo:** id, cod, cid, lote_odoo, qtd_ajuste, acao. **Aprovar com hash de listar-onda anterior bloqueia se ajustes foram alterados** (anti-replay). Workflow: listar → revisar → aprovar IMEDIATAMENTE.
- **G-PRE-08 Idempotencia do propor:** DELETE+INSERT zera+reinit. Re-rodar gera novos IDs → hash muda → re-listar antes de aprovar.
- **G-PRE-09 Onda 2 depende de Onda 5:** Onda 2 (TRANSFERIR_FB_CD residual) pressupoe que os lotes alvo do CD ja foram criados pela Onda 5. Executar 09b_executar_pre_etapa ANTES de scripts da Onda 2.
- **G-PRE-10 Executor v9 (orchestrator C3 macro):** capina 09b em `app/odoo/estoque/orchestrators/pre_etapa_executor.py`. Compoe Skills 1 (PURO via `ajustar_quant` criar_se_faltar=True com guard delta_esperado) + 2 (POS/NEG via `transferir_quantidade_para_lote_v2` com delta_esperado propagado). Mantem auditoria via OperacaoOdooAuditoria + paralelizacao (`--max-workers`). Pre-cond: ajustes status='APROVADO' (rodar `aprovar-onda` antes). Score: Skill 6 cobre ciclo completo planejar→propor→aprovar→executar.

## Exemplo (caso real: gerar Onda 5 ponta-a-ponta)

```bash
SK=.claude/skills/planejando-pre-etapa-odoo/scripts/planejar_pre_etapa.py

# 3. Revisar Excel (humano)

# decide: aceita ou refaz planilha de inventario fisico?

# 4. Propor (banco local)

# → 4 acoes INSERT em ajuste_estoque_inventario status=PROPOSTO

# → hash: "abc123def456...", valor_total: "32956127.45"

   --hash abc123def456... --usuario rafael --confirmar

python "$SK" --modo executar-onda --company-id 4 --limite 1 --usuario rafael --confirmar
# canary 1 produto → revisar Odoo → sub-piloto 10:
python "$SK" --modo executar-onda --company-id 4 --limite 10 --usuario rafael --confirmar
# bulk paralelo (todos APROVADOS):
python "$SK" --modo executar-onda --company-id 4 --max-workers 5 --usuario rafael --confirmar
```

## Outras variantes (sub-fluxos)

- **4.1.a — Preview antes de regenerar planilha:** rodar `planejar --dry-run` apos cada nova planilha de inventario fisico para avaliar impacto (total_pos/total_neg/total_residual_fb_cd) sem tocar disco.
- **4.1.b — Re-aprovar apos correcao:** se alguem alterou ajustes manualmente, hash diverge — refazer planejar+propor+listar+aprovar.
- **4.1.c — Onda 6 (FB futura):** `--company-id 1 --complementar none --confirmar`. Sem FB→FB residual (FB nao tem complementar; positivos puros cobrem deficits).
- **4.1.d — Subset de produtos (debug):** `--cods 210030325,4310177` restringe processamento.
- **4.1.e — Executar Onda APROVADA (NOVO v9 — orchestrator C3):**
   1. Pre-cond inviolavel: passos 1-6 do fluxo 4.1 rodados (ajustes status='APROVADO').
   2. **Canary OBRIGATORIO** (1 produto, dry-run): `--modo executar-onda --company-id 4 --limite 1`.
   3. Revisar JSON output (campos `contadores`, `produtos[0].pos_results/neg_results/puro_results`).
   4. **Canary real**: `--limite 1 --confirmar`. Verificar DIRETO no Odoo o quant tocado.
   5. **Sub-piloto** (10 serial): `--limite 10 --confirmar`.
   6. **Bulk paralelo**: `--max-workers 5 --confirmar`. CUIDADO: > 5 sobrecarrega Odoo XML-RPC.
   7. Auditar pos-execucao: `OperacaoOdooAuditoria` tem registros com pipeline_etapa='ONDA_5_PRE_ETAPA' por ajuste. `ajuste_estoque_inventario.status` migra para 'EXECUTADO' OU 'FALHA' (com `erro_msg`).
   8. **Retomada de FALHAs**: re-rodar `--modo executar-onda` so reprocessa APROVADO (FALHA fica de fora ate operador alterar manualmente para APROVADO).
   9. **Retomada single produto** (debug): `--cod-produto X --confirmar` processa apenas 1.
