# Fluxo 2.2 — Realocar saldo intra-empresa (sem NF)

**Quando:** o usuário pede para MOVER saldo de estoque DENTRO da mesma empresa, SEM emitir NF. Subcasos: (a) lote→lote na mesma location (consolidar MIGRAÇÃO num lote canônico, padronizar grafias, substituir lote); (b) location→location com o mesmo lote (FB/Estoque → FB/Indisponível, Indisponível → Estoque, Pré-Produção → Estoque). Operação reversível por outra transferência. 2 inventory adjustments compostos via Skill 1 (`ajustar_quant`); cada passo herda `delta_esperado` (regra inviolável 11).

**Galho da árvore (subagente):** `2 Estoque (sem NF) → 2.2 realocar saldo`. Esta folha cobre a transferência ATÔMICA. Para orquestração de PLANILHA com N linhas (D010/D012/D013), o HOST itera esta folha 1× por linha (a folha NÃO é o orquestrador).

---

## Premissas (pesquisar+validar ANTES de qualquer write)

1. **Empresa única por chamada.** `--empresa ∈ {FB, CD, LF}` resolve `company_id` e a `location_id` default via `app/odoo/estoque/_utils.py:resolver_empresa`. Inter-empresa exige NF — usar fluxo 1.x (`faturando-odoo` + `escriturando-odoo`).
2. **Produto:** `--cod` (default_code) resolve `product.product` via `_utils.resolver_produto`. **1 ativo esperado** — se 0 ou >1, parar e perguntar. `tracking == serial` é bloqueado pela skill (`BLOQUEADO_SERIAL`).
3. **Modo de operação (XOR — escolher 1):**
   - **MODO A — lote→lote (mesma location):** `--lote-origem A --lote-destino B`. Default location = principal da empresa (FB=8/CD=32/LF=42). Override via `--local`.
   - **MODO B — location→location (mesmo lote):** `--loc-origem L1 --loc-destino L2 [--lote N]`. Mesmo lote nos 2 lados. `--lote` ausente = quant sem lote (lot_id=False). Útil para mover MIGRAÇÃO entre Estoque ↔ Indisponível.
4. **Quantidade:** `--qty FLOAT` SEMPRE positivo. A skill aplica `delta=-qty` na origem e `delta=+qty` no destino internamente.
5. **Lote MIGRAÇÃO (G022):** se o nome do lote bate com `is_migracao(nome)` (`MIGRAÇÃO`, `MIGRACAO`, `MIGRAÇAO` — 3 grafias), a skill consolida no de MAIOR saldo na location alvo. Para forçar consolidação em UMA grafia específica (caso `padronizar_migracao`), passe `--lot-id` direto (futuro — limitação atual documentada em `_validados/transferindo-interno-odoo/VALIDACAO.md`).
6. **Reserva ativa na origem:** por default RESPEITA reserva (`FALHA_REDUCAO` se `qty_apos < reservada`). Para resetar reserved fantasma: `--resetar-reserva-origem` (NÃO cancela picking — limpa cache stale). Para reserva LEGÍTIMA de picking ativo, primeiro chamar **fluxo 2.4 `operando-reservas-odoo`** (cirurgia/cancelamento).
7. **Saldo vivo:** entre dry-run e `--confirmar` o saldo pode mudar (operação viva). Rodar `--dry-run` LOGO ANTES do `--confirmar`. Premissa `delta_esperado` (= `±qty`) é propagada → divergências disparam `FALHA_DELTA_DIVERGENTE` (proteção CICLAMATO).

---

## Sequência (átomos componíveis; output de um alimenta input do próximo)

```
1) (HOST/orquestrador) decidir MODO da operação e validar args:
   - MODO A: lote-origem + lote-destino (mesma loc; `--local` opcional)
   - MODO B: loc-origem + loc-destino (mesmo lote ou sem lote)
   - qty > 0; modos mutuamente exclusivos

2) transferindo-interno-odoo --dry-run
     args (MODO A):
        --cod <COD> --empresa <FB|CD|LF> [--local <id>]
        --qty <Q>
        --lote-origem <A>  --lote-destino <B>
        [--resetar-reserva-origem]
        [--tolerancia-delta T]   # default 0.001

     args (MODO B):
        --cod <COD> --empresa <FB|CD|LF>
        --qty <Q>
        --loc-origem <L1>  --loc-destino <L2>
        [--lote <N>]              # mesmo nos 2 lados; ausente = sem lote
        [--resetar-reserva-origem]
        [--tolerancia-delta T]

     → output: {modo:'dry-run', chave:{...}, resultado:{
          status, qty_transferida, lot_id_origem, lot_id_destino,
          reducao_origem:{status, qty_antes, qty_apos, ajuste_aplicado, ...},
          aumento_destino:{status, qty_antes, qty_apos, ajuste_aplicado, ...},
          tempo_ms
        }}

   - Status terminais OK: DRY_RUN_OK · EXECUTADO (quando --confirmar)
   - Falhas: FALHA_PRODUTO, FALHA_LOTE, FALHA_LOCAL, BLOQUEADO_SERIAL,
     FALHA_REDUCAO (com sub-status: FALHA_QUANT_VAZIO/FALHA_QUANT_NEGATIVO/
     FALHA_RESERVADO/FALHA_DELTA_DIVERGENTE), FALHA_AUMENTO, FALHA_ODOO.

3) (HOST) consolidar plano:
   - Apresentar: empresa, produto, modo, qty, origem→destino, qty_antes/apos
   - Se FALHA_REDUCAO/FALHA_AUMENTO: mostrar sub-status + erro
   - NÃO confirmar sem OK explícito.

4) transferindo-interno-odoo --confirmar
   - Mesmos args do passo 2 + `--confirmar`.
   - Verificar status final = EXECUTADO.
   - Em falha durante a confirmação: a reducao_origem pode TER OCORRIDO (idempotente em retry — checar
     `reducao_origem.status` antes de re-tentar).

5) (HOST) Verificar resultado DIRETO no Odoo (regra inviolável 6):
   - search_read stock.quant para origem e destino — confirmar qty pós-operação.
   - Em discrepância: NÃO assumir que falhou — verificar logs e o estado real.
```

## Gotchas do fluxo (codificados como invariantes no service `transfer.py`)

- **G021 (lot_id de empresa errada)** — `resolver_lote_origem/destino` SEMPRE filtra `company_id`. Lot de outra empresa estoura "Empresas incompativeis" no Odoo. Bug histórico de `transferir_lote.py` (2026-05-20: 11 aumentos falharam).
- **G022 (2 lotes MIGRAÇÃO/produto)** — wildcard das 3 grafias. Resolve no de MAIOR saldo na location alvo. Se nenhum tem saldo, retorna o primeiro existente. Se nenhum existe (destino MIGRAÇÃO), cria canônico `'MIGRAÇÃO'`. Caso `padronizar_migracao` (consolidar 2 GRAFIAS específicas) NÃO é coberto — requer `--lot-id` (futuro).
- **G027 (reserved_quantity vem de saída)** — `--resetar-reserva-origem` zera reserved ANTES do ajuste. NÃO cancela picking. Default RESPEITA reserva (proteção). Para reserva órfã/negativa específica, usar Skill 2.4 [[2.4-cancelar-reserva-orfa]] ANTES desta.
- **G028 (consolidar_move_lines)** — herdado da Skill 1 (`ajustar_quant`). Não afeta este fluxo se origem é quant simples.
- **G002 (`lot.name` search `=` instável)** — `StockLotService.buscar_por_nome` usa `in [valor]` + fallback. Herdado.
- **G_proxy_vazio** — `P-15/05` (literal) E `None` ambos mapeiam para `lot_id=False` (quant sem lote). Para MODO B com lote='P-15/05' a skill funciona; para MODO A com destino='P-15/05', a skill LEVANTA `ValueError` (use MODO B ou ajustar_quant diretamente).
- **G-TRANSFER-01 (criar_se_nao_existe retorna tuple)** — bug detectado em `transferir_fluxo_c` e `executar_fluxo_b_vivas` (ad-hocs). A skill NUNCA replica esse bug — `resolver_lote_destino` desempacota corretamente.
- **action_apply_inventory infla quant NEGATIVO** — herdado de `ajustar_quant`. Se aumento destino for em quant negativo, a Skill 1 protege via `validar_nao_negativar`. Para destino negativo intencional, usar fluxo 2.5 (picking).
- **delta_esperado** propagado a CADA chamada interna a `ajustar_quant` (regra inviolável 11). Se há divergência (qty_origem mudou desde dry-run), `FALHA_REDUCAO` com sub-status `FALHA_DELTA_DIVERGENTE`. Re-rodar dry-run.
- **Worktree sem `.env`** — para rodar dry-run/confirmar local, carregar `ODOO_*` da árvore original (sem `DATABASE_URL`).
