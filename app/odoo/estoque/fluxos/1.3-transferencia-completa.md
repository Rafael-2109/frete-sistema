# Fluxo 1.3 — Transferência completa (saída SEFAZ-OK + entrada escriturada)

**Quando:** preciso fazer uma operação inter-company end-to-end:
- Emitir NF de SAÍDA na company origem → SEFAZ autoriza → chave 35 dígitos
- Escriturar a entrada (DFe ou XML manual) na company destino → invoice posted
- Saldo do estoque consolidado em ambos os lados (saída na origem + entrada no destino)

Este é o **fluxo canônico de inventário inter-company** — todas as direções da MATRIZ_INTERCOMPANY (8 ações) seguem esta composição. Quase nunca rodamos só 1.1.1 (saída pura sem entrada) — exceto em sessões de recovery onde a entrada foi escriturada manualmente off-line.

**Composição:** este fluxo COMPÕE 2 sub-fluxos:
1. **1.1.1** — Saída pura até SEFAZ-OK (Skill 8 ATÔMICA L2 — 5 átomos sobre `account.move`)
2. **1.2.x** — Entrada escriturada no destino (Skill 7 ABRANGENTE — 10 átomos sobre DFe + account.move)
   - `1.2.1` (caminho A) se DFe chegou via SEFAZ automaticamente no destino
   - `1.2.2` (caminho B) se DFe ausente após timeout → criar manualmente via XML da SAÍDA

**Decisão A vs B é AUTOMÁTICA** dentro do átomo Skill 7 `executar_fluxo_l3_1_2_x` via `buscar_dfe(chave_nfe, company_destino)`.

**Operações cobertas (todas direções da MATRIZ_INTERCOMPANY):**

| Ação | Origem→Destino | Caminho típico 1.2.x |
|------|----------------|----------------------|
| INDUSTRIALIZACAO_FB_LF | FB→LF | **B** (DFe sentido reverso NUNCA vem via SEFAZ — sempre upload XML) |
| PERDA_LF_FB | LF→FB | A (DFe sincroniza FB) |
| DEV_LF_FB | LF→FB | A |
| DEV_CD_LF | CD→LF | A |
| DEV_LF_CD | LF→CD | A |
| DEV_FB_LF | FB→LF | A (suportado v27+ S4 — pendente canary) |
| TRANSFERIR_FB_CD | FB→CD | A |
| TRANSFERIR_CD_FB | CD→FB | A |

**Galho da árvore (subagente):** `1 NF inter-company → 1.3 transferência completa (saída + entrada)`.

**Quando NÃO usar esta folha:**
- Só saída sem entrada → fluxo 1.1.1 (folha terminal).
- Só entrada (DFe já SEFAZ-OK; saída foi feita externamente) → fluxo 1.2.1.
- Só criar DFe manual no destino sem ter feito a saída em sessão atual → fluxo 1.2.2.
- Compra externa de fornecedor (DFe não emitido por nós) → DELEGAR a `gestor-recebimento`.

**Responsabilidade:** este fluxo (L3) DECIDE quando invocar cada sub-fluxo. Nenhum átomo individual conhece esta composição — sempre passa pelo wrapper de cada sub-fluxo.

## Premissas (pesquisar+validar ANTES de qualquer write)

1. **Pré-cond da SAÍDA** (idem 1.1.1):
   - Picking de SAÍDA já em `state='done'` (Skill 5 ETAPA B)
   - Sub-skill C5 `auditando-cadastro-fiscal-odoo` retorna `pode_faturar=True`
   - Direção mapeada em `CONSTANTS_FLUXO_L3_POR_COMPANY_DESTINO` (v27+ S4: FB+CD+LF)
   - `confirmar_sefaz=True` para real-run

2. **Pré-cond da ENTRADA** (idem 1.2.1/1.2.2):
   - `account.move` de SAÍDA com `state='posted'` + `chave_nfe` válida (passo 5 de 1.1.1)
   - Para caminho A: DFe correspondente já disponível no destino via webservice SEFAZ (lookup automático em passo 1 de 1.2.x)
   - Para caminho B: XML da SAÍDA disponível em `account.move.l10n_br_xml_aut_nfe` (auto-populado pela SEFAZ)

3. **Direção mapeada nos dois lados:**
   - `CONSTANTS_FLUXO_L3_POR_COMPANY_DESTINO[company_destino]` retorna constants (v27+ S4: 3 companies)
   - `L10N_BR_TIPO_PEDIDO_POR_ACAO[acao_decidida]` mapeia `{dfe, po}` (v27+ S4: 8 ações)
   - Se não mapeado → reportar `NAO_SUPORTADA_V20` ANTES de transmitir SEFAZ (irreversível).

4. **Lote por ML mapeado** — para passo de `preencher_lotes_picking` em 1.2.x, mapping de lote por produto vem dos `AjusteEstoqueInventario.lote_destino` agregados. Fallback `INV-{cod}-{HOJE}`.

## Sequência (composição de 2 sub-fluxos)

```
FASE 1 — SAÍDA (fluxo 1.1.1):

1.1.1-passo-1  (opcional) Skill 8 validar_invoice_constants
1.1.1-passo-2  Skill 8 liberar_faturamento  →  dispara robo CIEL IT
1.1.1-passo-3  Skill 8 polling_invoice      →  retorna invoice_id de saida
1.1.1-passo-4  Skill 8 validar_invoice_pos_robo  →  G029+G007+G034
1.1.1-passo-5  Skill 8 transmitir_sefaz     →  IRREVERSIVEL; chave_nfe + autorizado

PONTO DE NAO RETORNO: SEFAZ autorizou. Saida persistida no banco.
  → Mesmo se Fase 2 falhar, podemos resumir mais tarde via `--modo resume`.

FASE 2 — ENTRADA (fluxo 1.2.x — caminho A ou B automatico):

1.2.x-passo-1  Skill 7 buscar_dfe(chave_nfe, company_destino)
                 → se encontrado=True E status='pendente' → caminho A
                 → se encontrado=False (apos timeout) → caminho B
                 → se status='processado' → IDEMPOTENT_DONE (sessao anterior ja fez)
                 → se INDUSTRIALIZACAO_FB_LF → forcar caminho B (DFe nunca chega)

CAMINHO A (DFe via SEFAZ):
1.2.1-passo-1.5  Skill 7 alinhar_dfe_lines_company(dfe_id, company_destino)
                   → fix B-V23-1 generalizado v25+ F2a
                   → batch write dfe.line.company_id = company_destino
1.2.1-passo-2  Skill 7 escriturar_dfe       →  l10n_br_tipo_pedido='compra' (DFe)
1.2.1-passo-3  Skill 7 gerar_po_from_dfe    →  fire-and-poll PO+picking nativo
1.2.1-passo-4  Skill 7 preencher_po         →  team_id+payment_term+picking_type+
                                                 l10n_br_tipo_pedido='<derivado>'(PO)
1.2.1-passo-5  Skill 7 confirmar_po         →  state='purchase' + invoice draft
1.2.1-passo-6  Skill 5 preencher_lotes_picking  →  mapping_lote_por_ml
1.2.1-passo-6.5 G023 force company_id no picking + moves (v25+ F2b)
1.2.1-passo-7  Skill 5 validar              →  state='done'
1.2.1-passo-8  Skill 7 criar_invoice_from_po  →  invoice draft (ou idempotente)
1.2.1-passo-9  (caller) account.move.action_post  →  state='posted'

CAMINHO B (DFe manual via upload XML SAIDA):
1.2.2-passo-0  Skill 7 criar_dfe_a_partir_do_invoice_saida(invoice_saida_id, company_destino)
                 → extrai XML de l10n_br_xml_aut_nfe + cria DFe destino

                    batch write dfe.line.company_id pos-poll)
1.2.2-passos-2 a 9 IDÊNTICOS ao caminho A (escriturar_dfe etc.)

PONTO FINAL: ambos os lados consolidados.
  - Saida: account.move state='posted', SEFAZ autorizado, picking state='done'
  - Entrada: account.move state='posted', PO state='purchase', picking state='done'
  - Saldo do estoque consolidado: -qty na origem + +qty no destino
  - AjusteEstoqueInventario.fase_pipeline='F5f_ENTRADA_OK' + status='EXECUTADO'
```

## Gotchas do fluxo

- **Atomicidade SEFAZ x DB local** (CRITICAL-1 v17): se passo 1.1.1-5 (SEFAZ) OK mas commit DB falha, saída autorizada NÃO é re-transmitida. Operador investiga.
- **Idempotência tripla** — cada átomo codifica seu próprio idempotency check. Resume `--modo resume --apenas-etapa F` continua de onde parou sem duplicar.
- **AP6 RESOLVIDO PARCIAL v24+ + S1 opt-in v27+** — Skill 8 ATÔMICA L2 (5 átomos `account.move`) separada de orchestrator C3 `inventario_pipeline`. Esta folha 1.3 COMPÕE Skill 8 (saída) + Skill 7 (entrada) via FLUXOS L3 — NUNCA via inline.
- **G011 timing CIEL IT** — robô CIEL IT cria invoice de saída em 3-30min (passo 1.1.1-3); robô cria PO em 3-5min madrugada / >30min pico (passo 1.2.x-3). Recovery via `--modo resume`.
- **G016 SSL drop** — todos os átomos com polling longo (1.1.1-3 e 1.2.x-3) têm `commit_resilient` + retry.
- **B-V23-1 + F2a v25+** — dfe.line.company_id herdado do XML SAÍDA (FB) precisa ser sobrescrito para company_destino (LF). Codificado em `criar_dfe_a_partir_do_invoice_saida` (caminho B) E em novo átomo `alinhar_dfe_lines_company` (caminho A v25+).
- **B-V23-2 v23.5+** — PO.line.account_id da company FONTE → resolver equivalente da company DESTINO via novo helper `resolver_account_id_por_company`. Codificado intra-átomo `gerar_po_from_dfe`.
- **F3a v25+ — tipos diferentes em DFe vs PO** — `dfe='compra'` UNIVERSAL (destrava action_gerar_po_dfe) + `po=<derivado MATRIZ>` (vira CFOP correto). Mapeamento `L10N_BR_TIPO_PEDIDO_POR_ACAO` v27+ S4 cobre 8 ações.
- **F4 v25+ team_id STATIC LF=143** — para destino LF, team fixo (Rafael) não derivado via G039. FB+CD ainda G039 dinâmico (v27+ S4 — pendente canary p/ decidir STATIC).
- **G023 force company picking+moves** (v25+ F2b) — picking nativo gerado via `action_gerar_po_dfe` pode sair com `location_id=4 Parceiros/Fornecedores` em vez de estoque destino. Hook força `stock.picking.company_id` + `stock.move.company_id = company_destino` no passo 6.5.
- **G-DFE-PURCHASE-FISCAL-ID-STALE** (descoberta cirurgia AVULSO_FRASCO v24+) — `dfe.purchase_fiscal_id` apontando para PO cancelada bloqueia `action_processar_arquivo_manual`. Skill 7 caminho B não trata automaticamente — exige cirurgia manual.
- **G-PO-NATIVA-SEM-PICKING** (descoberta cirurgia AVULSO_FRASCO v24+) — quando produto não tem `l10n_br_tipo_produto` cadastrado, `button_confirm` cria PO mas `_create_picking` não dispara. Sub-skill C5 v24+ check `tipo_produto_ausente` JÁ detecta. **Rodar pre-flight C5 ANTES desta folha**.

## Composição via orchestrator C3 `inventario_pipeline`

O orchestrator C3 LEGACY já roda esta composição inteira via pipeline A-F. Esta folha 1.3 DOCUMENTA semanticamente o fluxo, mas a execução prática é via:

```bash
python -m app.odoo.estoque.orchestrators.inventario_pipeline \
    --ciclo INVENTARIO_2026_05 \
    --etapas A,B,C,D,E,F \
    --usar-skill8-atomica-v25 \    # v27+ S1: ETAPAs C+D via Skill 8 ATOMICA L2 (fluxo 1.1.1)
    --usar-fluxo-l3-v19 \           # v20+ S3: ETAPAs E+F via FLUXO L3 1.2.x (caminho A ou B)
    --confirmar \
    --confirmar-sefaz \
    --pular-pre-flight  # exclua se quer rodar sub-skill C5 antes
```

Ambas opt-in podem coexistir. Default OFF preserva legacy.

Mapeamento ETAPAS A-F → fluxos:
| ETAPA | Conteúdo | Sub-fluxo (opt-in flags) |
|-------|----------|---------------------------|
| A | Skill 2 transferência interna pré-faturamento | (não modelado em fluxo L3 — átomo direto) |
| B | Skill 5 criar+validar picking saída inter-company | (não modelado em fluxo L3 — átomos diretos) |
| C | Skill 8 polling_invoice + validar_invoice_pos_robo | **fluxo 1.1.1 passos 3+4** (via `--usar-skill8-atomica-v25`) |
| D | Skill 8 transmitir_sefaz | **fluxo 1.1.1 passo 5** (via `--usar-skill8-atomica-v25`) |
| E | Skill 7 escriturar entrada (caminho A ou B) | **fluxo 1.2.x** (via `--usar-fluxo-l3-v19` — pendente FB destino v28+) |
| F | Skill 7 entrada destino LF (caminho A ou B) | **fluxo 1.2.x** (via `--usar-fluxo-l3-v19` — LF validado canary v20+) |

## Composição alternativa (Python direto — sem orchestrator)

Para casos isolados onde queremos rodar a transferência completa em 1 chamada Python (sem usar o orchestrator C3 LEGACY):

```python
from app.odoo.estoque.scripts.faturamento import FaturamentoInvoiceService
from app.odoo.estoque.scripts.picking import StockPickingService
from app.odoo.estoque.orchestrators.inventario_pipeline import (
    FaturamentoPipelineExecutor,  # para usar executar_fluxo_l3_1_2_x
)
from app.odoo.utils.connection import get_odoo_connection

odoo = get_odoo_connection()
picking_svc = StockPickingService(odoo=odoo)
svc_skill8 = FaturamentoInvoiceService(odoo=odoo, picking_svc=picking_svc)
executor = FaturamentoPipelineExecutor()  # para invocar executar_fluxo_l3_1_2_x

# Pre-cond: picking 12345 ja em state='done'; ajustes 111,112 em F5b

# === FASE 1 — SAIDA (fluxo 1.1.1) ===
r2 = svc_skill8.liberar_faturamento(
    picking_id=12345, ajuste_ids=[111, 112],
    ciclo='INVENTARIO_2026_05', dry_run=False, confirmar=True,
)
r3 = svc_skill8.polling_invoice(
    picking_id=12345, ajuste_ids=[111, 112],
    ciclo='INVENTARIO_2026_05', timeout_s=1800, dry_run=False,
)
invoice_saida_id = r3['invoice_id']

r4 = svc_skill8.validar_invoice_pos_robo(
    invoice_id=invoice_saida_id, ajuste_id_primeiro=111,
    perfil='inventario-inter-company',
    ciclo='INVENTARIO_2026_05', dry_run=False, confirmar=True,
)
r5 = svc_skill8.transmitir_sefaz(
    invoice_id=invoice_saida_id, ajuste_ids=[111, 112],
    ciclo='INVENTARIO_2026_05', dry_run=False, confirmar_sefaz=True,
)
chave = r5['chave_nfe']

# === PONTO DE NAO RETORNO: SEFAZ autorizou ===
# Mesmo se Fase 2 falhar, recovery via --modo resume cobre.

# === FASE 2 — ENTRADA (fluxo 1.2.x via executar_fluxo_l3_1_2_x) ===
# Decide caminho A vs B automaticamente
constants_lf = executor._resolver_constants_fluxo_l3(
    acao_decidida='INDUSTRIALIZACAO_FB_LF',  # ou outra acao
    company_destino=5,  # LF (v27+ S4: 1=FB, 4=CD, 5=LF)
)
assert constants_lf is not None

# Build lotes_data (passo 6 1.2.x)
lotes_data = [
    {'product_id': 12345, 'lote_nome': 'INV-103000011-20260527', 'quantidade': 10.0},
    {'product_id': 67890, 'lote_nome': 'INV-103000022-20260527', 'quantidade': 20.0},
]
# F1b v25+: lote_default fallback apenas para subprodutos auto-criados Odoo
lote_default = f'INV-FALLBACK-20260527'

r_fluxo = executor.executar_fluxo_l3_1_2_x(
    invoice_id_saida=invoice_saida_id,
    lotes_data=lotes_data,
    lote_default=lote_default,
    dry_run=False,
    **{k: v for k, v in constants_lf.items() if not k.startswith('_')},
)
assert r_fluxo['status'] == 'FLUXO_OK'
caminho = r_fluxo['caminho']  # 'A' ou 'B'
invoice_entrada_id = r_fluxo['invoice_id']  # PO -> invoice draft criada
# Posting final do invoice de entrada e responsabilidade do caller
```
