# Fluxo 2.2.j — Planilha (cod, qty) → Indisponivel em LOTE (orquestrador alto-nivel)

**Quando:** o usuario tem uma planilha com 10+ cods e quantidades a mover para `{emp}/Indisponivel`, sem saber qual lote origem usar para cada cod. A regra eh: "drena saldo VIVO do produto (locs internas exceto Indisp) ate a qty solicitada; consolida no destino em lote MIGRAÇÃO POR PRODUTO". Demanda real 2026-05-25 (158 cods FB).

**Galho da arvore:** `2 Estoque (sem NF) → 2.2 realocar saldo → 2.2.j para-indisp em LOTE`. Folha composta sobre `2.2.i (modo C atomico)`. Ad-hocs equivalentes: `mover_migracao_para_indisponivel.py` (1 lote MIGRAÇÃO so) + `ajuste_fb_cd_indisponivel.py` (planilha 4-col De-Para). Esta folha NAO substitui: cobre o caso COD+QTY (sem De-Lote), com origem descoberta via Skill 9.

---

## Premissas (validar ANTES de qualquer write)

1. **Empresa unica por chamada.** `--empresa ∈ {FB, CD, LF}`. Destino `{emp}/Indisponivel` resolvido via `LOCAIS_INDISPONIVEL[company_id]`.
2. **Planilha** (`--planilha CSV_PATH` ou `--cods "C1=Q1,..."` inline). Schema: `cod, qty, [nome]`. Decimal PT-BR (`,`) ou EN (`.`).
3. **Politica de ordem** (`--politica`): default `MIGRACAO_FIRST_FIFO` — drena MIGRAÇÃO primeiro, depois lotes reais em ordem lex de nome. Alternativas: `FIFO`, `MAIOR_SALDO`.
4. **Locs origem** (`--locs-origem`): default = `LOCS_ORIGEM_INTERNAS_POR_COMPANY[company]`. Para FB: `[8, 48, 4066, 4067, 4068, 27458]` (Estoque + Pos-Prod + Pre-Prod Vidro/Manual/Balde/Salmoura).
5. **Reserva ativa** (`--resetar-reserva-origem`): defensivo, zera `reserved_quantity` ANTES do ajuste. NAO cancela picking — apenas limpa cache stale. Para reserva LEGITIMA de picking ativo, considere fluxo 2.6 ANTES de rodar este orquestrador.
6. **Lote destino**: default `'MIGRAÇÃO'` (consolidador). Resolvido POR PRODUTO via `lot_svc.criar_se_nao_existe` (G031 — NAO eh universal por company).
7. **qty_solicitada = quantidade a MOVER** (delta de transferencia). Nao eh "saldo final" nem "saldo atual" — eh `delta=-qty` na origem + `delta=+qty` no destino.

---

## Sequencia (compoe Skill 2 modo C N vezes; output de um cod alimenta o proximo)

```
1) (orquestrador interno) carregar planilha + validar args:
   - Schema (cod, qty); decimal PT-BR/EN
   - --empresa resolve company_id + locs_origem default
   - --politica E POLITICAS_VALIDAS
   - qty > 0 por linha (linhas invalidas pulam com aviso stderr)

2) Para cada linha (cod, qty) — processa SEQUENCIALMENTE:
   2.1) resolver_produto(odoo, cod) → product_id
        Falha: FALHA_PRODUTO (cod nao existe). Proximo cod.

   2.2) consultando-quant-odoo._listar_quants_origem(
            product_id, company_id, locs_origem) → List[quant_dict]
        Filtra: quantity>0, lot_id not null, location_id IN locs_origem.
        Enriquece com _lote_name (1 read em lote — sem N+1).
        Vazio: FALHA_SEM_QUANT. Proximo cod.

   2.3) _ordenar_quants_origem(quants, politica)
        - MIGRACAO_FIRST_FIFO: (0 se MIGRAÇÃO else 1, nome_lote_lex, -quantity)
        - FIFO: (nome_lote_lex, -quantity)
        - MAIOR_SALDO: -quantity

   2.4) Greedy distribute:
        qty_falta = qty_solicitada
        for q in quants_ordenados:
            if qty_falta <= 0: break
            qty_disp = q.quantity if --resetar-reserva-origem else q.available
            if qty_disp <= 0:
                quants_pulados.append({motivo: 'sem saldo livre'})
                continue
            qty_p = min(qty_disp, qty_falta)
            try:
                res = transferindo-interno-odoo.transferir_para_indisponivel(
                    product_id, company_id,
                    lot_id_origem=q.lot_id,
                    qty=qty_p,
                    location_id_origem=q.location_id,
                    nome_lote_destino='MIGRAÇÃO',
                    resetar_reserva_origem=...,
                    tolerancia_delta=0.001,
                    dry_run=...,
                )
                transferencias.append({lot_id_origem, qty_pretendida=qty_p,
                                       qty_movida=res.qty_transferida, resultado: res})
                if status OK (DRY_RUN_OK | EXECUTADO):
                    qty_falta -= qty_p (dry) ou res.qty_transferida (real)
                else (FALHA_REDUCAO/FALHA_AUMENTO):
                    # Estado parcial em FALHA_AUMENTO: nao decrementa qty_falta,
                    # continua tentando outros quants.
                    pass
            except ValueError as e:
                # Pre-cond do atomo (ex.: lot_id_origem == lot_id_destino MIGRACAO)
                # NAO eh falha do cod inteiro — pula o quant e segue greedy.
                quants_pulados.append({motivo: f'atomo ValueError: {e}'})
                continue

   2.5) Sumario por cod:
        status = atingiu_total ? DRY_RUN_OK/EXECUTADO_TOTAL :
                                 DRY_RUN_PARCIAL/EXECUTADO_PARCIAL
        Append em resultados global.

3) (orquestrador externo) consolidar plano:
   - sumario macro (cods_total, cods_por_status, qty_solicitada/movida/nao_movida)
   - Apresentar pendencias do CSV (cods parciais/falhas para resolucao manual)

4) (orquestrador externo) Verificar resultado DIRETO no Odoo (regra inviolavel 6):
   - Consultando-quant-odoo (Skill 9) para cods do canary/sub-piloto.
   - Confirmar saldo destino aumentou + saldo origem reduziu.
```

## Gotchas do fluxo (codificados em distribuir_para_indisponivel)

- **Lote MIGRAÇÃO em loc origem (cod ja parcialmente consolidado)**: o atomo `transferir_para_indisponivel` levanta `ValueError` quando `lot_id_origem == lot_id_destino` (raio especial — o MIGRAÇÃO de FB/Estoque eh o MESMO stock.lot.id que o MIGRAÇÃO de FB/Indisponivel, pois `stock.lot` eh por produto — G031).
  - **v12**: o helper agora aplica AUTOMATICAMENTE fallback Modo B (`transferir_entre_locations`) mantendo o mesmo lote MIGRACAO, movendo origem → Indisp. Deteccao DUPLA: substring matching da msg + filtro semantico (`is_migracao(lote_name)`) — protege contra matching errado se atomo mudar a mensagem.
  - Output: `transferencias[].resultado._fallback_modo_b=True` + `_fallback_motivo`. Caso real 4310176 (1 un MIGRACAO em FB/Estoque + 1093 un lotes reais) → cobertura 100% via fallback (era 99.9% pre-v12).
  - Se fallback Modo B TAMBEM falhar (raro — ex.: reserva legitima no quant Modo B nao consegue burlar), quant vai para `quants_pulados[]` com motivo composto `modo C + fallback Modo B falharam`.
- **Ordem entre variantes MIGRACAO/MIGRAÇÃO**: lex puro do nome — `MIGRACAO` (sem cedilha, 'C' < 'Ç') vem ANTES de `MIGRAÇÃO` (com cedilha). Determinismo OK.
- **Saldo total < qty_solicitada**: status `DRY_RUN_PARCIAL`/`EXECUTADO_PARCIAL`. `qty_movida = saldo_disponivel_total`, `qty_nao_movida = qty_solicitada - saldo_disponivel_total`. Pendencia documentada em `--csv-pendencias`.
- **FALHA_AUMENTO em modo real**: estado PARCIAL — origem reduzida, destino nao creditado. `transferencias[i].resultado.qty_reduzida_origem` informa o debito parcial efetivado. O helper NAO decrementa qty_falta (segue tentando outros quants). Rollback via `ajustar_quant`+qty no quant origem (Skill 1).
- **`--resetar-reserva-origem` em PROD**: zera `reserved_quantity` ANTES do ajuste mas NAO cancela picking. Se o picking estiver ATIVO (ML viva), o `action_assign` futuro vai re-reservar — pode causar surpresa. Para reserva legitima, usar fluxo 2.6 ANTES.
- **`--cleanup-pos-bulk` (v12 — NOVO)**: apos o bulk, lista quants em FB exceto Indisp dos cods processados com transferencias executadas; aplica Skill 2.4 `zerar_residual` em `reserved<0` (com GUARD MO ativa via Skill 9 — pula quants com MLs vivas para evitar zerar reserva legitima) + Skill 1 `--valor-absoluto 0` em `qty<0` (saldos negativos de manual_consumption). Resultado em `payload.cleanup_pos_bulk` + CSV opcional `--csv-cleanup PATH`. Respeita `--dry-run`/`--confirmar`. Eleva exit code do CLI se cleanup falhar. **Resolve a licao v11** (cleanup precisou ser manual via 2 scripts separados).
- **CSV de pendencias**: 1 linha por cod com status != OK_TOTAL. Inclui qty_movida, qty_nao_movida, status, erro, produto_id, quants_disponiveis. Insumo para a sessao seguinte (resolver manual).
- **Sequencial** (sem paralelizacao): 158 cods FB em ~50s dry-run; estimativa 30-50 min em real (4-5s por action_apply_inventory × 493 transferencias). Sem --max-workers por hora (atomico — risco de race condition em quants do mesmo produto).

## Decisao: capinar ad-hoc vs novo orquestrador?

Esta folha NASCEU de demanda real (158 cods FB) que NAO cabia diretamente em:
- `mover_migracao_para_indisponivel.py` — so opera lote MIGRAÇÃO (qualquer cod).
- `ajuste_fb_cd_indisponivel.py` — exige De-Local + De-Lote por linha (planilha 4-col com schema diferente).
- Modo C atomico — exige `--cod --lote --qty` por chamada (sem politica de selecao de lote).

Solucao: helper composto sobre Modo C atomico que DESCOBRE lote origem via Skill 9 (read) + distribui qty greedy. Compatibilidade total com Modo C — apenas adiciona camada de orquestracao. **NAO substitui** os ad-hocs acima (cobrem casos diferentes).
