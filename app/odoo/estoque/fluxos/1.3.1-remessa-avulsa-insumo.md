# Fluxo 1.3.1 — Remessa avulsa de insumo inter-company (sem ciclo de inventário)

**Quando usar:** preciso remeter insumo direto (matéria-prima `tipo_produto='01'`, **sem explodir BoM**) de uma company origem para uma company destino **fora de ciclo de inventário** (`AjusteEstoqueInventario` NÃO criado, NÃO obrigatório). Caso-alvo: `INDUSTRIALIZACAO_FB_LF` avulsa — ex. remetendo componentes isolados de FB para LF.

**Operações cobertas:**
- `INDUSTRIALIZACAO_FB_LF` avulsa (FB→LF, CFOP 5901, fp 25)
- Qualquer direção inter-company de insumo direto fora do ciclo `inventario_pipeline`

**Quando NÃO usar esta folha:**
- Com ciclo de inventário (ajustes `AjusteEstoqueInventario` existem) → fluxo **1.3** (transferência completa com-ciclo; orchestrator C3 `inventario_pipeline`).
- Remessa com explosão de BoM (PA precisa de `e2e_remessa_criar.py` + BoM) → fora de escopo desta folha; ver driver `docs/industrializacao-fb-lf/RUNBOOK_PILOTO_4870112.md` para o piloto completo.
- Só escriturar entrada (saída já feita) → fluxo **1.2.1** (caminho A) ou **1.2.2** (caminho B).
- Só emitir NF de saída sem entrada → fluxo **1.1.1**.
- DFe de fornecedor externo → DELEGAR `gestor-recebimento`.

**Responsabilidade:** este fluxo (L3) ORIGINA diretamente os átomos (sem orchestrator C3 legacy). Compõe Skill 5 (picking) + Skill 8 (SEFAZ) + Skill 7 (entrada). `AjusteEstoqueInventario` é OPCIONAL (auditoria via `ajuste_ids`) — **não é pré-requisito**.

**Galho da árvore (subagente):** `1 NF inter-company → 1.3 transferência completa → 1.3.1 remessa avulsa de insumo (sem ciclo)`.

---

## Premissas (pesquisar+validar ANTES de qualquer write)

1. **Saldo disponível na origem** — executar ETAPA 0 (fluxo 2.2) antes se o saldo estiver em localização incorreta (ex.: `FB/Estoque/MIGRAÇÃO` → já em `{origem}/Estoque/{lote}` para picking de saída). Validar com `consultando-quant-odoo`.

2. **Pré-flight fiscal** — sub-skill C5 `auditando-cadastro-fiscal-odoo --perfil inventario` retorna `pode_faturar=True` para todos os produtos (NCM + barcode + weight válidos). Qualquer falha aborta antes de criar o picking.

3. **Lote de destino definido** — nome do lote que nascerá na company destino (ex.: `P-02/06`). NÃO é o mesmo lote da origem (G-ENT-6 — lotes inter-company são criados na company destino via C3).

4. **Direção mapeada em `CONSTANTS_FLUXO_L3_POR_COMPANY_DESTINO`** — confirmar que a company destino tem constants (v27+ S4: companies FB=1, CD=4, LF=5). Consultar `app/odoo/estoque/orchestrators/inventario_pipeline.py` para a constante `CONSTANTS_FLUXO_L3_POR_COMPANY_DESTINO` antes de transmitir SEFAZ (irreversível).

5. **Caminho de entrada SEMPRE B para `INDUSTRIALIZACAO_FB_LF`** — o DFe que chega via SEFAZ na LF é resumo vazio (G-ENT-2); a decisão A vs B em `executar_fluxo_l3_1_2_x` é forçada para B (C2). Os demais pares de direção podem usar caminho A automático.

---

## Sequência (átomos componíveis; output de um alimenta input do próximo)

```
PRÉ-FLIGHT (sempre antes de write):

0)  [pré-cond] fluxo 2.2 — realocar saldo se necessário (ETAPA 0 interna)
    Skill 2 transferindo-interno-odoo --dry-run    # só se saldo não está em {origem}/Estoque/{lote}

0b) sub-skill C5 auditando-cadastro-fiscal-odoo --perfil inventario --cods <lista>
    → deve retornar pode_faturar=True em todos os produtos
    → bloqueio imediato se qualquer produto com NCM/weight/barcode inválido

FASE 1 — SAÍDA INTER-COMPANY (Skill 5 + Skill 8):

1)  Skill 5 criar_picking_inter_company  →  picking_id, state='assigned'
    args: produto+qty+lote (por move.line), partner_id=destino,
          picking_type=saída inter-company (pt53 p/ FB→LF),
          location_id={origem}/Estoque, location_dest_id=Em Trânsito ({transit_id})
    dry-run: revisar moves + partner_id + local
    confirmar: criar picking real + validar (button_validate)

2)  [G-ENT-1] Skill 5 cancelar picking-companheiro "Transferir TERCEIROS"
    → o server action 1899 auto-cria um picking-companheiro (picking_terceiro_id)
       no contexto da company ORIGEM (FB=1 para FB→LF)
    → Skill 5 cancelar picking_terceiro_id EM CONTEXTO allowed_company_ids=[company_origem]
       (senão: "sem acesso leitura a stock.picking" — company_id=FB, mas ctx=LF)
    → o trânsito {origem_transit} mantém a qty; drena no passo G5 do recebimento LF

3)  Skill 8 liberar_faturamento  →  dispara robô CIEL IT
    args: picking_id, ajuste_ids=None (OPCIONAL aqui — C1),
          ciclo=None (avulso, sem ciclo inventário)
    dry-run: verificar que picking está done antes de chamar
    confirmar: action_liberar_faturamento → aguardar robô criar NF (~90s)

4)  Skill 8 polling_invoice  →  invoice_id (account.move state='posted', situacao_nf='rascunho')
    args: picking_id, ajuste_ids=None, timeout_s=1800
    → robô CIEL IT cria a invoice de saída em 3-90s; G011 timing: retry up to 30min pico

5)  Skill 8 validar_invoice_pos_robo  →  G029+G007+G034 checks
    args: invoice_id, perfil='inventario-inter-company'
    → aborta se invoice inválida (tipo, fiscal_position, valor)

⚠️  PONTO DE NÃO RETORNO — SEFAZ (IRREVERSÍVEL):

6)  Skill 8 transmitir_sefaz  →  chave_nfe (35 dígitos), situacao_nf='autorizado'
    args: invoice_id, ajuste_ids=None (C1 — idempotência intra-Odoo via account.move),
          dry_run=False, confirmar_sefaz=True
    → idempotência: se situacao_nf=='autorizado' ou chave_nfe populada → IDEMPOTENT_SKIP (C1)
    → fail-closed avulso: sem ajuste_ids, qualquer estado indeterminado → FALHA_ESTADO_INDETERMINADO (C1)
    *** CONFIRMAÇÃO EXPLÍCITA DO USUÁRIO OBRIGATÓRIA ANTES DE CHAMAR ***
    *** APRESENTAR A NF DRAFT AO USUÁRIO (invoice_id, valor, CFOP, parceiro) ANTES ***

FASE 2 — ENTRADA DESTINO (Skill 7):

7)  Skill 7 executar_fluxo_l3_1_2_x  →  fluxo_status='FLUXO_OK', invoice_entrada_id
    args:
      invoice_id_saida=<invoice_id do passo 6>
      lotes_data=[{'product_id': X, 'lote_nome': '<lote_destino>', 'quantidade': Y}, ...]
      lote_default='<lote_destino>'  # fallback p/ subprodutos auto-criados Odoo
      company_destino=<id>           # ex.: 5 para LF
      # constants derivadas de CONSTANTS_FLUXO_L3_POR_COMPANY_DESTINO[company_destino]:
      l10n_br_tipo_pedido_dfe='serv-industrializacao'  # G-ENT-3: escriturar_dfe rejeita 'compra' (canary 627348 usou serv-industr)
      l10n_br_tipo_pedido_po='serv-industrializacao'   # PO/invoice (canary usou ambos = serv-industr)
      team_id=143                    # STATIC LF (F4 v25+); FB/CD: G039 dinâmico (None)
      payment_term_id=2791           # 'A VISTA' universal
      picking_type_id=19             # LF; 1=FB, 13=CD (discovery v27+ S4)
      payment_provider_id=38         # 'SEM PAGAMENTO' (G029)
    → caminho B forçado p/ INDUSTRIALIZACAO_FB_LF (C2: buscar_dfe detecta resumo_vazio)
    → C3 fixa company do lote: StockLotService resolve/cria lote_destino em company_destino
    → preencher_lotes_picking com company_destino=<id> (arg obrigatório — G-ENT-6)
    dry-run: revisar PO + picking + lotes antes de validar

8)  [G-ENT-5] Se picking de entrada NÃO auto-gera (FALHA_PASSO_7_SEM_PICKING):
    → diagnóstico: listar rotas do produto (search product.supplierinfo + route_ids)
    → causa provável: conflito regras `buy` FB×LF (route 5 + route 133 simultâneas)
    → fallback SEGURO: Skill 5 criar picking manual Vendors(4)→{destino_estoque}
                       vinculado à PO (moves com purchase_line_id)
                       + preencher_lotes_picking company_destino=<id>
                       + validar picking
    → fallback ARRISCADO (alterar rotas): NÃO fazer sem autorização explícita do usuário
    → se não há fallback seguro claro: PARAR e reportar diagnóstico

9)  Skill 7 criar_invoice_from_po  →  invoice draft (ou idempotente)
    (interno ao executar_fluxo_l3_1_2_x — passos 1.2.x-8+9)

10) account.move.action_post  →  invoice entrada state='posted'
    (caller — passo 1.2.x-9 do fluxo 1.2.x)

AUDITORIA FINAL:

11) consultando-quant-odoo --modo quants --empresa <destino> --produto <cod>
    → confirmar saldo em {destino}/Estoque/{lote_destino} = qty remetida
    → confirmar saldo origem baixou correspondentemente
```

## Gotchas do fluxo

| # | Gotcha | Causa | Solução codificada |
|---|--------|-------|-------------------|
| **G-ENT-1** | picking-companheiro "Transferir TERCEIROS" é da company ORIGEM | Server action 1899 auto-cria picking no contexto da company que faz a remessa (FB=1 para FB→LF); ler/cancelar exige `allowed_company_ids=[origem]` senão "sem acesso leitura a stock.picking" | Passo 2 desta sequência — cancelar em contexto origin explícito |
| **G-ENT-2** | DFe via SEFAZ é resumo vazio (0 linhas, sem XML) para INDUSTRIALIZACAO_FB_LF | DFe de industrialização nunca chega populado — só resumo `l10n_br_status='06'`; `action_gerar_po_dfe` sem linhas → PO vazia | C2: `buscar_dfe` qualifica `resumo_vazio`; `executar_fluxo_l3_1_2_x` força caminho B |
| **G-ENT-5** | PO de entrada confirmada mas picking NÃO auto-gera | Conflito de regras `buy` FB×LF (route 5 "Comprar-FB" + route 133 "Comprar-LF" coexistem) → `_create_picking` não dispara | Passo 8: diagnóstico + fallback picking manual Vendors→destino vinculado à PO |
| **G-ENT-6** | "Empresas incompatíveis" ao validar picking entrada com lote da origin | Lote criado na remessa (company FB=1) não pode ser reutilizado pela LF; `company_id` de lote com estoque é imutável | C3: `preencher_lotes_picking company_destino=<id>` resolve/cria lote na company destino |
| **G-ENT-3** | `escriturar_dfe` rejeita `tipo='compra'` para DFe de industrialização | Whitelist de `escriturar_dfe` (`escrituracao.py:1388-1392`) aceita `serv-industrializacao`/`transf-filial`/`retorno`/`outro`/`industrializacao`/`perda`/`dev-industrializacao` — **NÃO** `'compra'`; passar `'compra'` → erro `tipo_pedido_invalido` | Usar `l10n_br_tipo_pedido_dfe='serv-industrializacao'` (canary REAL 627348 — RUNBOOK §0.7). PO/invoice (`preencher_po`, whitelist `escrituracao.py:1972-1975`) também aceita `serv-industrializacao` |

**Nota âncora `AjusteEstoqueInventario`:** este fluxo NÃO depende de `AjusteEstoqueInventario`. A idempotência intra-Odoo da Skill 8 `transmitir_sefaz` (C1) é baseada no próprio `account.move` (`situacao_nf`/`chave_nfe`) — a âncora é **OPCIONAL** (auditoria via `ajuste_ids`), não pré-requisito. Sem `ajuste_ids`, a fase de rastreabilidade no banco local (D8.3) é pulada; a proteção anti-dupla-transmissão SEFAZ permanece íntegra via intra-Odoo.

**Atomicidade SEFAZ × DB local:** se SEFAZ (passo 6) OK mas commit falha — saída autorizada NÃO é re-transmitida. Idempotência intra-Odoo detecta na próxima tentativa. Retomar pela Fase 2 (passo 7+) sem repetir Fase 1.

## Como o gestor executa (composição Python direto — sem orchestrator C3 legacy)

Para o caso avulso (sem `inventario_pipeline`), o gestor monta a execução diretamente:

```python
from app.odoo.estoque.scripts.faturamento import FaturamentoInvoiceService
from app.odoo.estoque.scripts.picking import StockPickingService
from app.odoo.estoque.orchestrators.inventario_pipeline import FaturamentoPipelineExecutor
from app.odoo.utils.connection import get_odoo_connection

odoo = get_odoo_connection()
picking_svc = StockPickingService(odoo=odoo)
svc_skill8 = FaturamentoInvoiceService(odoo=odoo, picking_svc=picking_svc)
executor = FaturamentoPipelineExecutor()

# === FASE 1 — SAÍDA ===

# 1) Picking saída (Skill 5) — criar + validar
# [dry-run primeiro: confirmar antes de executar]
picking_id = <id do picking saída, já criado e done>

# 2) Cancelar picking-companheiro "TERCEIROS" (G-ENT-1)
# picking_terceiro_id = picking saída.picking_terceiro_id
# Skill 5 cancelar em ctx allowed_company_ids=[company_origem]

# 3-5) liberar → poll → validar
r3 = svc_skill8.liberar_faturamento(
    picking_id=picking_id,
    ajuste_ids=None,       # avulso — sem ciclo (C1 torna opcional)
    ciclo=None,
    dry_run=False, confirmar=True,
)
r4 = svc_skill8.polling_invoice(
    picking_id=picking_id,
    ajuste_ids=None,
    timeout_s=1800, dry_run=False,
)
invoice_saida_id = r4['invoice_id']

r5 = svc_skill8.validar_invoice_pos_robo(
    invoice_id=invoice_saida_id,
    ajuste_id_primeiro=None,   # opcional (C1)
    perfil='inventario-inter-company',
    dry_run=False, confirmar=True,
)

# 6) SEFAZ — IRREVERSÍVEL (apresentar NF ao usuário antes)
# *** confirmar_sefaz=True SÓ com "go" explícito do usuário ***
r6 = svc_skill8.transmitir_sefaz(
    invoice_id=invoice_saida_id,
    ajuste_ids=None,      # C1: idempotência via account.move (situacao_nf)
    dry_run=False, confirmar_sefaz=True,
)
chave = r6['chave_nfe']

# === PONTO DE NÃO RETORNO: SEFAZ autorizou ===
# Fase 2 pode ser retomada sem repetir Fase 1 (idempotência C1).

# === FASE 2 — ENTRADA ===

# Constantes da company destino (LF=5):
# CONSTANTS_FLUXO_L3_POR_COMPANY_DESTINO[5] — consultar orchestrator
constants_destino = {
    'company_destino': 5,                           # LF
    'l10n_br_tipo_pedido_dfe': 'serv-industrializacao',  # G-ENT-3: escriturar_dfe rejeita 'compra' (ver ⚠️ abaixo)
    'l10n_br_tipo_pedido_po': 'serv-industrializacao',   # canary 627348 usou ambos = serv-industr
    'team_id': 143,                                 # STATIC LF (F4 v25+)
    'payment_term_id': 2791,                        # A VISTA
    'picking_type_id': 19,                          # LF
    'payment_provider_id': 38,                      # SEM PAGAMENTO
}

lotes_data = [
    {'product_id': 105000002, 'lote_nome': 'P-02/06', 'quantidade': 30.56},
    {'product_id': 105000044, 'lote_nome': 'P-02/06', 'quantidade': 5.36},
]

# 7) Entrada destino (Skill 7) — caminho B forçado (C2) + lote company destino (C3)
r7 = executor.executar_fluxo_l3_1_2_x(
    invoice_id_saida=invoice_saida_id,
    lotes_data=lotes_data,
    lote_default='P-02/06',
    dry_run=False,
    **{k: v for k, v in constants_destino.items() if not k.startswith('_')},
)
assert r7['status'] == 'FLUXO_OK'
invoice_entrada_id = r7['invoice_id']

# 10) Posting do invoice de entrada
# account.move.action_post via XML-RPC ou via executor interno

# 11) Auditoria final
# consultando-quant-odoo --modo quants --empresa LF --produto 105000002
```

> **Nota:** `executor.executar_fluxo_l3_1_2_x` com `company_destino=5` força **caminho B** (C2 detecta DFe-resumo via `buscar_dfe`). Para `INDUSTRIALIZACAO_FB_LF` passe `l10n_br_tipo_pedido_dfe='serv-industrializacao'` (o canary REAL 627348 usou esse valor; `escriturar_dfe` rejeita `'compra'`). O arg `company_destino` é `kwarg-only` obrigatório — não omitir. Ver nota do splat em [folha 1.3](1.3-transferencia-completa.md) §"Composição Python direto".

> ✅ **RESOLVIDO (Rafael 2026-06-02):** o mapeamento do pipeline (`L10N_BR_TIPO_PEDIDO_POR_ACAO['INDUSTRIALIZACAO_FB_LF']['dfe']`, `inventario_pipeline.py:3317`) foi corrigido de `'compra'` para `'serv-industrializacao'`, alinhado com o `escriturar_dfe` (rejeita `'compra'`), o canary 627348 e a decisão fiscal. Entrada FB→LF usa `'serv-industrializacao'` em DFe + PO + fatura. Validação final do `action_gerar_po_dfe` com `serv-industrializacao` ocorre no canary REAL. Continue passando `l10n_br_tipo_pedido_dfe='serv-industrializacao'` explícito no avulso (coerente com o mapeamento).
