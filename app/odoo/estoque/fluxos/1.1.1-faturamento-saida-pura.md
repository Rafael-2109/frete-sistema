# Fluxo 1.1.1 — Faturamento de saída pura (NF inter-company até SEFAZ-OK)

**Quando:** preciso emitir NF de SAÍDA inter-company para a SEFAZ autorizar, **sem entrada** no destino. Tipicamente este fluxo é **composto pelo fluxo 1.3** (transferência completa = saída + entrada). Como folha isolada, usa-se quando o destino vai escriturar a entrada manualmente em sessão separada, ou quando o destino está fora do nosso grupo (rara — todas as direções da MATRIZ_INTERCOMPANY são intra-grupo).

**Operações típicas:**
- INDUSTRIALIZACAO_FB_LF (saída FB → LF)
- PERDA_LF_FB / DEV_LF_FB (saída LF → FB)
- DEV_CD_LF / DEV_LF_CD (saída CD↔LF)
- TRANSFERIR_FB_CD / TRANSFERIR_CD_FB (saída FB↔CD)

**Pré-requisito de COMPOSIÇÃO:** ajustes em `AjusteEstoqueInventario` já com `status='APROVADO'` + `fase_pipeline='F5a_PICKING_OK'` ou `F5b_PICKING_VALIDADO`. Picking de SAÍDA já criado + validado por Skill 5 (`criar_picking_inter_company` + `validar_picking_inter_company`) na ETAPA A+B do orchestrator C3 `inventario_pipeline`.

**Galho da árvore (subagente):** `1 NF inter-company → 1.1 só faturamento (saída) → 1.1.1 caso de inventário (a partir de stock.picking validado)`.

**Quando NÃO usar esta folha:**
- Saída de venda/cliente (não inter-company) → outro fluxo (não modelado nesta árvore).
- DFe de fornecedor (entrada de compras) → DELEGAR a `gestor-recebimento`.
- Apenas escriturar entrada de DFe SEFAZ-autorizada externamente → fluxo 1.2.1.
- Inventário completo (saída + entrada inter-company) → fluxo 1.3 (composição desta com 1.2.x).

**Responsabilidade:** este fluxo (L3) DECIDE quando invocar cada átomo da Skill 8 ATÔMICA L2. As skills L2 são átomas e não decidem composição. **Skill 7 (`escriturando-odoo`) NÃO participa deste fluxo** — Skill 7 cuida SOMENTE de entrada/escrituração; saída/SEFAZ é responsabilidade da Skill 8.

---

## Premissas (pesquisar+validar ANTES de qualquer write)

1. **Picking de SAÍDA em `state='done'`** — `stock.picking.id` validado pela Skill 5 atom `validar_picking_inter_company`. Validação: `consultando-quant-odoo --modo pickings --picking-id <ID>` confirma `state=done`.
2. **`account.move` de SAÍDA candidato** — gerado pelo robô CIEL IT após `action_liberar_faturamento` do picking. Pode não existir ainda (será criado pelo átomo 2 do fluxo).
3. **`fiscal_position_id` e `l10n_br_tipo_pedido` da SAÍDA** — derivados de `MATRIZ_INTERCOMPANY[acao]['fiscal_position_id'][(co_origem, co_destino)]` + `MATRIZ_INTERCOMPANY[acao]['l10n_br_tipo_pedido']`. Setados pelo orchestrator/service legado ao criar o picking (G034 fiscal_setup).
4. **Cadastro fiscal completo** — sub-skill C5 `auditando-cadastro-fiscal-odoo --perfil inventario` retorna `pode_faturar=True` (todos produtos do picking com NCM + barcode + weight válidos). Bloqueio em qualquer um aborta esta folha.
5. **Direção mapeada em `CONSTANTS_FLUXO_L3_POR_COMPANY_DESTINO`** — `_resolver_constants_fluxo_l3(acao, company_destino)` retorna `None` se direção não suportada (v27+ S4: 3 companies mapeadas — FB+CD+LF; demais retorna `None`).
6. **`confirmar_sefaz=True` em real-run** — SEFAZ é IRREVERSÍVEL. NF autorizada com chave 35 dígitos só cancela via processo formal 24h sem uso + declaração. Real-run sem flag explícita retorna `BLOQUEADO_SEM_CONFIRMAR_SEFAZ`.
7. **Playwright disponível** — host com Chromium + Odoo UI credentials (ODOO_USERNAME + ODOO_PASSWORD env). Sem isso, `transmitir_sefaz` retorna `FALHA_CONFIG` (D7 HARD_FAIL).

## Sequência (átomos componíveis; output de um alimenta input do próximo)

```
1) Skill 8 validar_invoice_constants --invoice-id <PRE_INVOICE_ID> \
        --constants-esperadas '{fiscal_position_id: X, l10n_br_tipo_pedido: Y, ...}'
   → output: {status: 'OK'|'FALHA_DIVERGENCIA', divergencias: {...}}
   - PRE-COND da SAÍDA: se invoice já existe pre-faturamento (raro — geralmente
     robô CIEL IT é quem cria), validar campos fiscais antes de disparar
     liberar_faturamento. Pode ser pulado se invoice ainda não existe.
   - Modo READ-only (não escreve no Odoo)

2) Skill 8 liberar_faturamento --picking-id <PICKING_ID> \
        --ajuste-ids <ID1,ID2,...>
   → output: {status: 'OK', picking_state: 'done'}
   - Delega Skill 5 LEGACY picking_svc.liberar_faturamento (codifica G019/G020)
   - Dispara robô CIEL IT: cria account.move de saída em 3-30min
   - Pre-cond: picking.state='done' (validado pela Skill 5 anteriormente)
   - NÃO retorna invoice_id (criação assíncrona)

3) Skill 8 polling_invoice --picking-id <PICKING_ID> \
        --ajuste-ids <ID1,ID2,...> \
        --timeout-s 1800 --poll-interval-s 40
   → output: {status: 'OK', invoice_id: int}
   - Fire-and-poll: busca account.move com ref=picking.name
   - G011 timing CIEL IT: 3-5min madrugada, 5-10min manhã, >2h pico
   - Timeout=1800s cobre 95% dos casos; recovery via `--modo resume --apenas-etapa C`

4) Skill 8 validar_invoice_pos_robo --invoice-id <INVOICE_ID> \
        --ajuste-id-primeiro <AID> \
        --perfil 'inventario-inter-company'
   → output: {status: 'OK', sub_etapas: {f5d5_*, f5d6_*, f5d7_*}}
   - Aplica G029 (payment_provider=38) + G007 (price_unit fallback) +
     G034 (DEV_* fiscal_setup) via `_invoice_helpers`
   - Idempotente: helpers checam estado pré-existente
   - Pre-cond: invoice em fase F5d_INVOICE_GERADA

5) Skill 8 transmitir_sefaz --invoice-id <INVOICE_ID> \
        --ajuste-ids <ID1,ID2,...> \
        --max-tentativas 15 --intervalo-retry 120 \
        --confirmar-sefaz  # IRREVERSIVEL
   → output: {status: 'OK', chave_nfe: '35260...', situacao_nf: 'autorizado'}
   - Playwright SEFAZ via UI Odoo (serial, 1 browser, 5-10min/NF)
   - Atualiza fase_pipeline='F5e_SEFAZ_OK' + chave_nfe + status='EXECUTADO'
     em TODOS ajustes do mesmo invoice (D-OPS-2b §7.5.2)
   - IDEMPOTENT_SKIP se invoice já em F5e_OK (anti-double-transmission)
   - D7 HARD_FAIL_CONFIG aborta batch (playwright_indisponivel etc.)
   - CRITICAL-1: commit POS-Playwright falha = NÃO conta sucesso
```

## Gotchas do fluxo

- **G016 SSL drop** — durante polling (passo 3) ou Playwright longo (passo 5), conexão pode cair. Átomo `polling_invoice` delega ao Skill 5 LEGACY que tem retry codificado; `transmitir_sefaz` faz `commit_resilient` antes E depois.
- **G011 timing CIEL IT** — passo 3 (polling_invoice) tem timeout 1800s. Acima disso → ABORTAR e recovery via `--modo resume --apenas-etapa C --usar-skill8-atomica-v25`.
- **G019/G020 picking state** — codificado intra-átomo `liberar_faturamento` (delega Skill 5 LEGACY). Re-lê state pós-`button_validate` e raise se != 'done'.
- **G029/G007/G034** — codificados em `validar_invoice_pos_robo` via `_invoice_helpers` (perfil V1='inventario-inter-company'). Outros perfis raise NotImplementedError.
- **D-OPS-2b §7.5.2** — `transmitir_sefaz` propaga `chave_nfe` para TODOS ajustes do mesmo invoice (mesmo se invoice mistura ajustes de produtos diferentes). Fix definitivo (filtrar por `account.move.line`) é TODO pós-canary.
- **D8.3 idempotência tripla** — atom `transmitir_sefaz` codifica skip por persistência (F5e_OK ou status=EXECUTADO em QUALQUER ajuste do invoice → skip todo o invoice).
- **CRITICAL-1 v17 commit POS-Playwright** — se SEFAZ autorizou mas commit DB falhou, NÃO conta sucesso (estado em memória não persistido). Operador investiga manualmente.
- **MED C-1/C-2 cstat+xmotivo** — átomo persiste `cstat` (rejeição SEFAZ) + `xmotivo` em falhas para audit fiscal.
- **CFOP NÃO hardcodar** — derivado pelo motor fiscal Odoo via `fiscal_position + l10n_br_tipo_pedido`. Atom NÃO seta `l10n_br_cfop_id` diretamente.
- **NUNCA cancelar NF autorizada via XML-RPC** — processo formal SEFAZ (24h, sem uso, declaração). Caso erro detectado pós-`transmitir_sefaz` OK: operador investiga, NÃO automatizar cancelamento.

## Composição típica (Python — orchestrator C3 ou caller direto)

```python
from app.odoo.estoque.scripts.faturamento import FaturamentoInvoiceService
from app.odoo.estoque.scripts.picking import StockPickingService
from app.odoo.utils.connection import get_odoo_connection

odoo = get_odoo_connection()
picking_svc = StockPickingService(odoo=odoo)
svc = FaturamentoInvoiceService(odoo=odoo, picking_svc=picking_svc)

# Pre-cond: picking 12345 ja em state='done' (criado+validado por Skill 5)
# Pre-cond: ajustes 111, 112 em fase F5b_PICKING_VALIDADO

# 1. (opcional) validar constants — pode pular se invoice ainda nao existe
# r1 = svc.validar_invoice_constants(invoice_id=..., constants_esperadas={...})

# 2. Liberar faturamento (dispara robo)
r2 = svc.liberar_faturamento(
    picking_id=12345, ajuste_ids=[111, 112],
    ciclo='INVENTARIO_2026_05',
    dry_run=False, confirmar=True,
)
assert r2['status'] == 'OK'

# 3. Polling do robo CIEL IT (3-30min)
r3 = svc.polling_invoice(
    picking_id=12345, ajuste_ids=[111, 112],
    ciclo='INVENTARIO_2026_05',
    timeout_s=1800, dry_run=False,
)
assert r3['status'] == 'OK'
invoice_id = r3['invoice_id']

# 4. Aplicar G029+G007+G034
r4 = svc.validar_invoice_pos_robo(
    invoice_id=invoice_id, ajuste_id_primeiro=111,
    perfil='inventario-inter-company',
    ciclo='INVENTARIO_2026_05',
    dry_run=False, confirmar=True,
)
assert r4['status'] in ('OK', 'OK_PARCIAL')

# 5. Transmitir SEFAZ (IRREVERSIVEL)
r5 = svc.transmitir_sefaz(
    invoice_id=invoice_id, ajuste_ids=[111, 112],
    ciclo='INVENTARIO_2026_05',
    max_tentativas=15, intervalo_retry=120,
    dry_run=False, confirmar_sefaz=True,
)
assert r5['status'] == 'OK'
chave = r5['chave_nfe']  # NF autorizada
```

## Composição via orchestrator C3 `inventario_pipeline` (opt-in v27+ S1)

O orchestrator C3 LEGACY já chama ETAPAs C+D do pipeline A-F com lógica inline equivalente (~500 LOC de `executar_etapa_c` + `executar_etapa_d`). A partir de v27+ S1, a flag `--usar-skill8-atomica-v25` faz ETAPAs C+D delegarem aos átomos 3, 4 e 5 da Skill 8 ATÔMICA L2 (este fluxo).

```bash
python -m app.odoo.estoque.orchestrators.inventario_pipeline \
    --ciclo INVENTARIO_2026_05 \
    --etapas A,B,C,D \
    --usar-skill8-atomica-v25 \
    --pular-pre-flight \
    --confirmar \
    --confirmar-sefaz
```

Default OFF preserva legacy. Migração para usar este fluxo via opt-in v25+ S1 codificada no orchestrator (helpers `_executar_etapa_c_via_skill8_atomica` + `_executar_etapa_d_via_skill8_atomica`).
