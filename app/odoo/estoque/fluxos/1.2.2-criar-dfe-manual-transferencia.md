# Fluxo 1.2.2 — Criar DFe a partir do XML da SAÍDA + escriturar (caminho B — DFe SEFAZ ausente)

**Quando:** a NF de SAÍDA já está SEFAZ-autorizada (chave 35 dígitos) MAS o DFe correspondente NÃO apareceu no Odoo destino via webservice SEFAZ (timing CIEL IT G011, ou direção sem sincronia DFe automática — caso típico: INDUSTRIALIZACAO_FB_LF nunca tem DFe automático). Como a NF é **nossa** (transferência interna FB↔LF↔CD), o XML autorizado JÁ EXISTE no `account.move` da SAÍDA (`l10n_br_xml_aut_nfe`) e podemos **upload-ar** esse XML no destino criando DFe + dispará-lo via `action_processar_arquivo_manual`. Daí em diante a sequência é IDÊNTICA ao caminho A (1.2.1).

**Não confundir com:** "criar picking de entrada manual" (tampão arquitetural AP2 deprecado v19+). Aqui fazemos upload do XML existente para criar DFe; o picking é gerado pelo Odoo automaticamente via PO confirmada.

**Responsabilidade**: este fluxo (L3) DECIDE invocar `criar_dfe_a_partir_do_invoice_saida` quando o caminho A não viabilizou. As skills L2 são átomas e não decidem composição. **Skill 8 (`faturando-odoo`) NÃO participa deste fluxo** — Skill 8 cuida SOMENTE de saída/SEFAZ; entrada é responsabilidade da Skill 7 + Skill 5.

**Galho da árvore (subagente):** `1 NF inter-company → 1.2 só entrada/escrituração → 1.2.1 inventário (DFe próprio)` com decisão `buscar_dfe` → `encontrado=False`.

**Quando NÃO usar esta folha:**
- DFe é de fornecedor externo (CTe, Compras) → **erro ao usuário**: não temos XML para criar DFe manualmente. DELEGAR a `gestor-recebimento` se for o caso.
- DFe veio normalmente via SEFAZ → ir para fluxo 1.2.1 (caminho A).
- Operação SEFAZ ainda não foi feita (NF não está autorizada) → ETAPA D do orchestrator Skill 8 deve ir antes.

## Premissas (pesquisar+validar ANTES de qualquer write)

1. **NF é nossa (origem é uma das 3 companies NACOM):** chave NF-e foi emitida por FB/CD/LF (CNPJ emitente ∈ {`18.467.441/0001-63` LF, `61.724.241/0001-69` FB, etc.}). Validar via `account.move` SAÍDA: `company_id` ∈ {1, 4, 5}.
2. **NF está SEFAZ-autorizada:** `account.move.l10n_br_situacao_nf == 'autorizado'` AND `state == 'posted'`. Skill 8 (faturando-odoo) já transmitiu.
3. **XML autorizado existe no invoice SAÍDA:** `account.move.l10n_br_xml_aut_nfe` não-vazio (base64). Garantido para NF SEFAZ-OK. Validação no átomo `criar_dfe_a_partir_do_invoice_saida` — se vazio, raise.
4. **DFe NÃO apareceu via SEFAZ no destino:** Skill 7 `buscar_dfe(chave_nfe, company_destino)` retornou `encontrado=False` após N tentativas (timeout polling). N depende da operação:
   - **INDUSTRIALIZACAO_FB_LF**: DFe **nunca** vem via SEFAZ (sentido reverso — não há l10n_br_dfe_entries automático). N=0 → caminho B direto.
   - **PERDA_LF_FB, DEV_LF_FB, TRANSFERIR_CD_FB**: DFe **geralmente** vem via SEFAZ. Tentar caminho A primeiro (N=3 tentativas, 30min cada); se nada veio, fallback caminho B.
5. **`l10n_br_tipo_pedido` da entrada**: lookup `MATRIZ_INTERCOMPANY[acao]['entrada'][(co, cd)]['l10n_br_tipo_pedido_entrada']` (idêntico a 1.2.1).
6. **`fiscal_position_id` da entrada**: idem lookup. Canary se `None` (DEV_LF_FB, DEV_LF_CD têm precedente fiscal incompleto).
7. **Mapeamento lote por ML**: idem 1.2.1 — `AjusteEstoqueInventario.lote_destino` agregados, fallback `'MIGRAÇÃO'`.
8. **Idempotência cross-folha**: rodar `buscar_dfe` primeiro mesmo neste caminho B — se DFe apareceu entre tentativas, redirecionar para 1.2.1 sem criar DFe duplicado.

## Sequência (átomos componíveis; output de um alimenta input do próximo)

```
1) Skill 7 buscar_dfe --chave-nfe <CHAVE> --company <DESTINO>
   → output: {encontrado: bool, dfe_id: int|None, status: str}

   - encontrado=False → seguir passo 2

2) (FLUXO) decidir: NF é nossa? (cnpj_emitente ∈ NACOM)
```

## Gotchas do fluxo

- **AP2 RECLASSIFICADO (v19+)** — antes desta folha, ETAPA F do orchestrator atual criava picking de entrada manual via Skill 5 v15a `criar_picking_entrada_destino_manual` (tampão arquitetural). Causa real: orchestrator SAÍDA estava criando picking de ENTRADA (violação fronteira). Solução: criar DFe via upload XML da SAÍDA + deixar Odoo gerar picking via PO. Tampão Skill 5 v15a permanece como museum vivo (deprecada) com docblock explicativo.
- **XML autorizado existe** — para NF SEFAZ-OK, `account.move.l10n_br_xml_aut_nfe` é populado pelo Odoo durante transmissão. Pré-cond garantida por Skill 8 ter rodado antes deste fluxo. Se vazio (caso anômalo), átomo raise FALHA_XML_VAZIO.
- **`criar_dfe_a_partir_do_invoice_saida` é só para NF nossa** — pré-cond do FLUXO (não da skill): `cnpj_emitente ∈ CNPJs_NACOM`. Skill apenas extrai XML do invoice_id passado; quem decide se invoice é nosso = FLUXO. Para CTe/Compras (XML externo), fluxo aborta antes de invocar a skill.
- **action_processar_arquivo_manual fire-and-poll** — o Odoo parseia XML em ~5-30s. Service externo (L919-922) chama action + poll por `l10n_br_status`/`l10n_br_situacao_dfe` até estabilizar. Padrão preservado no átomo.
- **Idempotência DFe duplicado** — átomo busca primeiro: existe DFe com mesma chave_nfe na mesma company? Se sim, retorna esse dfe_id sem criar duplicado (e re-upload XML se faltava — pattern L930-932 do service externo).
- **company do DFe NÃO via context** — service externo NÃO usa `allowed_company_ids`. Confia em `company_id` correto no `create` do DFe + usuário XML-RPC. Atomo segue mesmo padrão (vs minha proposta inicial errada com `ctx_force_company` no `gerar_po_from_dfe`).
- **G011 timing CIEL IT** — após `action_gerar_po_dfe` em DFe upload-ado, robô materializa PO em 3-5min/madrugada a >30min/pico. timeout_s=1800 default cobre 95% dos casos.
- **G016 SSL drop** — idem 1.2.1.
- **CFOP derivado** — mesma regra: `l10n_br_cfop_id` resolvido via fiscal_position + tipo_pedido. NÃO hardcodar.
- **Direções suportadas v19+ (subset)**: INDUSTRIALIZACAO_FB_LF (canônico — sempre B), TRANSFERIR_CD_FB, TRANSFERIR_FB_CD, PERDA_LF_FB, DEV_LF_FB, DEV_CD_LF, DEV_LF_CD, DEV_FB_LF. Validar contra `ACAO_PARA_DIRECAO` em `operacoes_fiscais.py`.

## Decisão arquitetural v19+

`criar_dfe_a_partir_do_invoice_saida` na Skill 7 substitui o tampão Skill 5 v15a. Razão: subir o XML da SAÍDA (que já existe — toda NF SEFAZ-OK tem `l10n_br_xml_aut_nfe`) mantém o motor Odoo no fluxo natural (DFe → PO → picking → invoice), preservando CFOP derivado, fiscal_position e demais invariantes fiscais. Criar picking direto (v15a) bypassava o motor fiscal e exigia hardcodar CFOP (G037 — caso degenerado que sai de cena após esta folha).

**Fronteira clara**:
- Skill 7 átomos = construir/configurar DFe + PO + invoice (1 objeto por átomo)
- Skill 5 átomos = picking (1 objeto)
- Skill 8 átomo (restrita C+D) = liberar_faturamento + polling invoice + SEFAZ. **NÃO entra neste fluxo de ENTRADA.**
- FLUXO L3 (esta folha) = decide ordem + decide caminho A vs B + decide quando invocar Skill 8 (para SAÍDA — caso `1.3-transferencia-completa.md` pendente)
- Orchestrator C3 = invoca FLUXOS L3 conforme caso (inventário, transferência, devolução, etc.)
