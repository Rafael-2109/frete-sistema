# Fluxo 2.6 — Tratar reserva ATIVA bloqueando transferência futura

**Quando:** antes de uma transferência (Skill 2 modo A/B/C), o saldo do quant origem está parcial ou totalmente RESERVADO por picking(s) ativo(s). Se prosseguir cego, a transferência falha (`UserError` Odoo "qty disponível < demanda") ou — pior — desencadeia comportamento imprevisto. Este fluxo INVESTIGA quem está reservando e oferece os **5 caminhos seguros** para liberar.

**Galho da árvore (subagente):** `2 Estoque (sem NF) → 2.6 tratar reserva bloqueia transferência (pré-Skill 2)`.

**Regra inviolável (gestor-estoque-odoo):** SEMPRE checar `reserved_quantity` via Skill 9 dos quants candidatos a DOAR ANTES de chamar Skill 2. SE `reserved > 0`, investigar pickings via fluxo 2.6 ANTES de prosseguir.

---

## Premissas (pesquisar+validar ANTES de qualquer write)

1. **Identificar quant candidato a doar** (origem da transferência):
   - Tipicamente o quant que a Skill 2 (modo A/B/C) vai reduzir.
   - Pode vir de planilha (cod + empresa + lote) ou de inspeção ad-hoc.
2. **Checar `reserved_quantity` real** via Skill 9 modo quants:
   ```bash
   python .claude/skills/consultando-quant-odoo/scripts/consultar_quants.py \
     --cods <cod> --empresas <emp> --com-lote <padrao>
   # Buscar campo "reserved" na saída
   ```
3. **Se reserved > 0** → seguir este fluxo. Caso contrário, ir direto para Skill 2.
4. **Identificar pickings reservando** via Skill 9 modo pickings (NOVO v7):
   ```bash
   python .claude/skills/consultando-quant-odoo/scripts/consultar_quants.py \
     --modo pickings --quant-ids <q1,q2,...>
   ```
   → retorna pickings DISTINCT com state, name, picking_type, origin, partner.
5. **Classificar cada picking encontrado**:
   - **state = `done`** → DEVOLVER (caminho B); preservação fiscal cuidadosa.
   - **state = `assigned/partially_available/confirmed/waiting`** → opções A/C/D/E:
     - **picking_type = OUT/Expedição** + **origin contendo "Devolução"** → caso fiscal complexo, **NUNCA cancelar sem consultar Fiscal**.
     - **picking_type = INT/Transferência Interna** sem origin + sem partner → provavelmente operação ad-hoc, candidato natural para CANCELAR (caminho A).
     - **picking_type = EMB/Separação Embalagem** + **origin = FB/OP/MANUAL/xxxxx** → MO ativa consumindo o lote; cancelar picking = furo em MO. Avaliar MO antes (Skill 4 `medir_consumo`).
     - **MLs órfãs** (`find_orphan_mls` retorna >0) → caminho E (cirurgia).
   - **OUTRO LOTE livre** com saldo suficiente → caminho D (mais seguro — não tocar reserva).

## 5 caminhos seguros para desreservar (tabela de decisão)

| Caminho | Comando | Quando usar | Risco | Reversível? |
|---|---|---|---|---|
| **A. Cancelar picking inteiro** | Skill 5 `--modo cancelar --picking-id <P>` OU Skill 2.4 `--cancelar-picking --picking-id <P>` | Picking sem valor fiscal (fantasma; operação abandonada; INT sem origem/partner) | IRREVERSÍVEL. Tratamento fiscal se NF emitida (consultar Fiscal). | ❌ |
| **B. Devolver picking** | Skill 5 `--modo devolver --picking-id <P>` | Picking state=`done` que precisa estornar saldo | Cria NF de devolução. Estorno fiscal pode ser necessário. | Parcial (via outra devolução) |
| **C. Desreservar mantendo picking** | Skill 2.4 `--unreserve-picking --picking-id <P>` (NOVO v7) | Operador quer liberar reserva mas manter picking para re-reservar com saldo atual | **RISCO conhecido (G_UNRESERVE_TRAVA)**: picking pode TRAVAR em assigned se Odoo re-reservar automaticamente. Verificar `n_mls_depois==0` no output. | ✅ (re-reserva via Odoo) |
| **D. Não desreservar, usar OUTRO lote** | Skill 2 `--lote-origem <ALT> --modo A` | Existe outro lote livre com saldo suficiente para a doação | Mais seguro — não toca reserva existente. Mas pode não haver lote alternativo. | ✅ |
| **E. Cirurgia em ML órfã** | Skill 2.4 `--ml-ids <ML1> --moves-writes "<M1>:<qty_OK>"` | Picking tem MIX de MLs OK + órfãs (`find_orphan_mls` >0); quer preservar MLs OK | Cobre APENAS MLs **órfãs** (quant zerado). NÃO aplica para reserva ATIVA em quant com saldo. | ❌ |

**Regra de seleção:**
- D primeiro (mais seguro, sem risco fiscal).
- E se há órfãs misturadas.
- A se picking é fantasma/sem valor.
- B se picking já em done com estorno necessário.
- C como último recurso (G_UNRESERVE_TRAVA).

## Gotchas do fluxo

- **G_UNRESERVE_TRAVA** (caminho C): `stock.picking.do_unreserve` retorna sem erro mas picking pode CONTINUAR em `state=assigned` se Odoo re-reservar automaticamente (depende de trigger). Sempre verificar `n_mls_depois==0` no output da Skill 2.4 `unreserve_picking`.
- **G030** (Skill 9): `stock.move.line.quant_id` é computed `store: False` — não filtrar por ele direto. A Skill 9 `listar_pickings_por_quant` resolve via tupla (product, lot, location, company) internamente.
- **Picking fiscal (OUT/Expedição com origin=Devolução)**: NUNCA cancelar (caminho A) sem consultar Fiscal — pode invalidar NF de devolução já emitida.
- **MO ativa consumindo lote** (EMB com origin=FB/OP/MANUAL): cancelar picking = furo em MO. Avaliar MO via Skill 4 `medir_consumo_mo` ANTES; se MO tem consumo>0, prefira caminho D (outro lote).
- **MLs órfãs misturadas**: caminho E exige `moves_writes` calculado por MOVE PARENT, não por ML (1 move pode ter 1 órfã + N OK; `product_uom_qty` novo = soma das MLs OK).
- **`reserved` negativo pos-cirurgia**: SEMPRE chamar `zerar_reserved_residual` após caminho E.
- **Caminho D é PREFERIDO** quando há lote alternativo livre — sem risco fiscal, sem mexer em reserva, atômico.
