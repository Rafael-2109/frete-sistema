# Fluxo 2.9 — Consulta de quants AO VIVO no Odoo (READ-only)

**Quando:** o pedido envolve consultar saldos/MLs/pickings AO VIVO no Odoo (não no DB local sincronizado). Tipicamente para **auditoria pós-WRITE** ("sobrou saldo após o ajuste?"), snapshots pontuais ("onde está o saldo do produto X?"), validação de premissas antes de WRITE.

**Galho da árvore (subagente):** `2 Estoque (sem NF) → 2.9 CONSULTA AO VIVO de quants/MLs/pickings`.

**Diferença vs `consultando-sql`:** aquela é DB LOCAL sincronizado (latência minutos/horas). Esta é Odoo XML-RPC AO VIVO (latência ~1-2s, estado real).
**Diferença vs `gestor-estoque-producao`:** aquele é READ-ONLY analítico (ruptura, projeção, agregados) sobre DB LOCAL. Esta é cirúrgica (filtros de quant/lote/local) AO VIVO.

---

## Premissas (pesquisar+validar ANTES de consultar)

1. **Identificar o escopo** da consulta:
   - **Por cods** (lista de default_codes) ou **por pids** (lista de product_ids).
   - **Por empresas** (FB/CD/LF) ou **por pares (cod, empresa)** específicos (para evitar produto-cartesiano).
   - **Por lote** (padrão ilike — ex.: 'MIGRA' captura MIGRACAO/MIGRAÇÃO).
2. **Definir filtros opcionais**:
   - `--excluir-indisp`: remove location Indisponivel (FB=31088, CD=31090, LF=31091).
   - `--only-principal`: limita a location principal (FB=8, CD=32, LF=42).
   - `--incluir-qty-zero`: incluir quants vazios (útil para descobrir MLs órfãs).
3. **Decidir granularidade do output**:
   - Detalhado (1 linha por quant) vs agregado por (cod, empresa).
   - Para listas grandes (≥100 cods): preferir `--agregar`.

---

## Sequência (1 átomo direto, sem dependências de WRITE)

```
1) (HOST) Identificar filtros (cods, empresas/pares, lote, locations_excluir).

2) (CALL) consultando-quant-odoo:
   python .claude/skills/consultando-quant-odoo/scripts/consultar_quants.py \
     [--cods c1,c2,...] [--empresas FB,CD] [--excluir-indisp] [--com-lote MIGRA] \
     [--only-principal] [--incluir-qty-zero] [--agregar] [--formato json|tabela]
   → output: {total_quants, quants:[...], agregado:{(cod,empresa):{...}}}

3) (HOST — opcional) Auditar N pares (cod, empresa) em alto nível:
   Via service direto (CLI ainda não exposto):
      svc.auditar_pares([(cod1,'FB'), (cod2,'CD'), ...])
   → classifica cada par em [totalmente_zerado, so_indisp, com_saldo_nao_indisp, sem_produto]
   Útil para auditoria pós-WRITE de planilha de ajustes.

4) (HOST) Interpretar o resultado:
   - "Sobrou saldo em loc !=Indisponivel?" → `qty_total > 0` em `--excluir-indisp`.
   - "Está só em Indisponivel?" → comparar 2 chamadas (com vs sem `--excluir-indisp`).
   - "Reservado negativo (estado fantasma)?" → buscar `reserved < 0` no output.
```

---

## Receitas (caso real -> args)

| Preciso de... | Args | Comentário |
|---|---|---|
| Saldo restante após ajuste em N produtos | `--cods c1,c2,... --excluir-indisp --agregar` | Caso real motivador (2026-05-23) |
| Quants em lote MIGRACAO em FB+CD | `--com-lote MIGRA --empresas FB,CD` | Diagnóstico fantasma |
| Saldo só na location principal | `--cods c1,c2 --only-principal` | Validar saldo "real" |
| MLs órfãs em quants zerados | `--incluir-qty-zero --excluir-indisp` (filtrar quants com reserved!=0 no output) | Pós-`--resetar-reserva` |
| Auditoria pares (cod, empresa) classificada | Service direto: `auditar_pares([...])` | Skill 9 átomo 2 |

---

## Gotchas do fluxo

- **`cods + empresas` é PRODUTO-CARTESIANO** — cada cod busca em CADA empresa. Para "pares específicos", use `pares_cod_empresa` (parâmetro do service) ou rode múltiplas queries.
- **`incluir_qty_zero=False` é o default** — quants vazios são excluídos por padrão (otimização). Passar `--incluir-qty-zero` para descobrir MLs órfãs (quant qty=0 reserved!=0).
- **Custo XML-RPC** cresce com N cods + N empresas + N quants encontrados. Para milhares: prefira `monitor/1_baixar_estoques.py` (snapshot batch para CSV).
- **Estado pode estar mudando**: query é "agora" — quants podem mudar entre chamada e ação subsequente. Para WRITE seguro, repetir query imediatamente antes.
