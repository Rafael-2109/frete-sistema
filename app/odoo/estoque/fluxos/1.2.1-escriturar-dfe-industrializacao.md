# Fluxo 1.2.1 — Escriturar DFe (caminho A — DFe disponível via SEFAZ)

**Quando:** a NF de SAÍDA já está SEFAZ-autorizada (chave 35 dígitos) E o DFe correspondente já apareceu no Odoo destino via webservice SEFAZ (`l10n_br_dfe_entries.document`). Caminho **canônico** de escrituração de entrada — Odoo deriva CFOP via `fiscal_position` + `l10n_br_tipo_pedido` (NÃO hardcodar CFOP).

**Operações típicas neste caminho A**: PERDA_LF_FB, DEV_LF_FB, DEV_CD_LF, TRANSFERIR_CD_FB, TRANSFERIR_FB_CD — direções onde o robô CIEL IT do destino sincroniza DFe automaticamente após SEFAZ-OK da SAÍDA.

**Direções que NÃO usam este caminho A**: INDUSTRIALIZACAO_FB_LF — DFe **nunca** vem via SEFAZ automaticamente neste sentido (não há sincronia DFe automática FB→LF). Operação sempre cai em **fluxo 1.2.2** (caminho B — DFe criado via upload do XML da SAÍDA).

**Galho da árvore (subagente):** `1 NF inter-company → 1.2 só entrada/escrituração → 1.2.1 qualquer operação onde DFe já chegou via SEFAZ`.

**Quando NÃO usar esta folha:**
- `buscar_dfe` retornou `encontrado=False` após timeout → ir para **fluxo 1.2.2** (caminho B — `criar_dfe_a_partir_do_invoice_saida`) **APENAS** se a NF é nossa (transferências internas FB↔LF↔CD); senão, erro ao usuário.
- DFe é de fornecedor externo (XML que não emitimos) → **DELEGAR a `gestor-recebimento`** (fluxo 1.2.2 da árvore de roteamento — NÃO confundir com esta folha 1.2.2-criar-dfe...).

**Responsabilidade**: este fluxo (L3) DECIDE quando invocar cada átomo da Skill 7 + Skill 5. As skills L2 são átomas e não decidem composição. **Skill 8 (`faturando-odoo`) NÃO participa deste fluxo** — Skill 8 cuida SOMENTE de saída/SEFAZ; entrada é responsabilidade da Skill 7 + Skill 5.

## Premissas (pesquisar+validar ANTES de qualquer write)

1. **Chave NF-e 35 dígitos** (`l10n_br_chave_nf`) — vinda do `account.move` da SAÍDA já SEFAZ-OK ou de `AjusteEstoqueInventario.chave_nfe`. Validação: regex `^\d{44}$`.
2. **Company destino** (`company_id_recebedor`): 1=FB, 4=CD, 5=LF. Vem da `MATRIZ_INTERCOMPANY[acao]['entrada'][(co, cd)]` — chave da entrada usa MESMA tupla da saída; entrada é escriturada no `company_destino`.
3. **`l10n_br_tipo_pedido` da entrada**: lookup em `MATRIZ_INTERCOMPANY[acao]['entrada'][(co, cd)]['l10n_br_tipo_pedido_entrada']`. Valores típicos:
   - `'serv-industrializacao'` (INDUSTRIALIZACAO_FB_LF → CFOP 1901 derivado)
   - `'retorno'` (PERDA_LF_FB → 1903 / DEV_*_* → 1949)
   - `'transf-filial'` (TRANSFERIR_*_* → 1151/1152)
   - `'outro'` (DEV_LF_FB / DEV_LF_CD canary fiscal)
4. **`fiscal_position_id` da entrada**: idem lookup. Pode vir `None` (canary fiscal sem precedente — bloquear ou avisar usuário).
5. **DFe disponível**: skill 7 `buscar_dfe(chave_nfe, company_id_recebedor)` retornou `encontrado=True` E `status=='pendente'` (ainda não escriturado). Se `status=='processado'` → idempotência (pular sequência, reportar `IDEMPOTENT_DONE`).
6. **Mapeamento lote por ML**: para cada produto da NF, qual lote destino aplicar no picking gerado. Vem dos `AjusteEstoqueInventario.lote_destino` agregados por `(product_id, cfop)`. Fallback: `'MIGRAÇÃO'`.
7. **`l10n_br_cfop_id` NÃO hardcodar**: motor fiscal Odoo deriva via `fiscal_position` + `l10n_br_tipo_pedido`. Validar pós-PO confirmada (operação viva — comparar com `cfop_esperado` da matriz para log).
8. **Saldo na origem**: verificar saldo positivo do produto+lote+location de origem antes de escriturar — picking gerado via DFe→PO precisa ter saldo reservável. Skill 9 `consultando-quant-odoo --modo quants`.

## Sequência (átomos componíveis; output de um alimenta input do próximo)

```
1) Skill 7 buscar_dfe --chave-nfe <CHAVE> --company <DESTINO>
   → output: {encontrado: True, dfe_id: int, status: 'pendente'|'processado', raw: {...}}
   - Se encontrado=False → ABORTAR esta folha; ir para fluxo 1.2.2 (criar DFe manual)
   - Se status='processado' → IDEMPOTENT_DONE (DFe já escriturado em sessão anterior; verificar PO/picking/invoice)

2) Skill 7 escriturar_dfe --dfe-id <ID> \
        --l10n-br-tipo-pedido <TIPO> \
        --fiscal-position-id <FP>
   → output: {escriturado: True, l10n_br_tipo_pedido: <TIPO>, fiscal_position_id: <FP>}
   - Configura o DFe para que action_gerar_po_dfe saiba como gerar PO
   - Pre-cond: dfe.state ∈ {'pendente', 'a_processar'} (não 'processado')

3) Skill 7 gerar_po_from_dfe --dfe-id <ID> \
        --poll-timeout-s 1800
   → output: {po_id: int, picking_id: int (auto-criado pelo Odoo), tempo_ms: int}
   - Chama action_gerar_po_dfe via XML-RPC; service externo NÃO usa `allowed_company_ids` no context
   - Company correta vem do `company_id` do próprio DFe + usuário XML-RPC (D-V19-2)
   - Fire-and-poll: dispara action e monitora purchase.order aparecer com origin=chave_nfe (timeout 1800s default)
   - G011 timing CIEL IT: robô pode demorar 3-5min (madrugada) a >30min (pico)
   - Picking de entrada é GERADO PELO ODOO automaticamente via PO

4) Skill 7 preencher_po --po-id <ID> \
        --team-id <TEAM> \
        --payment-term-id <PT> \
        --picking-type-id <PT_ID> \
        --l10n-br-tipo-pedido <TIPO>
   → output: {po_preenchida: True}
   - Preenche campos obrigatórios da PO que o robô CIEL IT não preencheu sozinho
   - Pre-cond: po.state in ('draft', 'sent')

5) Skill 7 confirmar_po --po-id <ID>
   → output: {state: 'purchase', invoice_status: 'no'|'to invoice'}
   - button_confirm na PO
   - Trigger Odoo: gera invoice draft (in_invoice) + valida picking auto-criado (move-se para state='assigned')

6) Skill 5 preencher_lotes_picking --picking-id <PICKING_ID> \
        --lote-default 'MIGRAÇÃO' \
        --mapping '{pid1: lote1, pid2: lote2}'
   → output: {mls_preenchidas: int, mls_skipped: int}
   - Atribui lote em cada stock.move.line do picking gerado
   - mapping_lote_por_ml: dict por product_id (vem do AjusteEstoqueInventario.lote_destino)
   - Fallback lote_default: 'MIGRAÇÃO' (consolidador inter-company padrão)
   - Pre-cond: picking.state='assigned'

7) Skill 5 validar --picking-id <PICKING_ID>
   → output: {state: 'done'}
   - button_validate via Skill 5 átomo existente
   - G019/G020 invariante codificada no service: re-lê state e raise se != 'done'

8) Skill 7 criar_invoice_from_po --po-id <ID>
   → output: {invoice_id: int, state: 'draft', move_type: 'in_invoice'}
   - Se PO já gerou invoice draft no passo 5: retorna esse invoice_id (idempotente)
   - Caso contrário: action_create_invoice via XML-RPC
   - NÃO posta — só cria draft. Posting é responsabilidade do orchestrator Skill 8 ETAPA D (SEFAZ — mas para ENTRADA não há SEFAZ, é só posting Odoo)

9) (HOST/orchestrator) postar invoice via account.move.action_post
   → output: {state: 'posted'}
   - Para entrada inter-company: NÃO emite NF-e (já é entrada — XML SEFAZ já veio externamente)
   - Apenas posting contábil no Odoo
```

## Gotchas do fluxo

- **G011 timing CIEL IT** — robô leva 3-5min madrugada / 5-10min manhã / >2h pico para action_gerar_po_dfe materializar PO. `timeout_s=1800` cobre 95% dos casos. Acima disso → ABORTAR e ir para fluxo 1.2.2 (criar DFe manual) se aplicável.
- **G016 SSL drop** — durante polling longo (passos 3 e 5), conexão pode cair. Wrapper de retry + keepalive (já codificado em `get_odoo_connection`). Re-fetch ajustes via `safe_session_get` se sessão SQLAlchemy detached.
- **D-V19-2 — `gerar_po_from_dfe` NÃO usa `allowed_company_ids` no context** — service externo `RecebimentoLfOdooService` confia em `company_id` do próprio DFe + usuário XML-RPC. Atomo v19+ segue mesmo padrão. Garantir `company_id` correto no DFe (passos 1 e 2) ANTES do passo 3.
- **CFOP derivado, não hardcodado** — `l10n_br_cfop_id` na PO/invoice é resolvido pelo motor fiscal via `fiscal_position` + `l10n_br_tipo_pedido`. NUNCA setar diretamente (sigue regra `MATRIZ_INTERCOMPANY` docstring + G037).
- **CFOP esperado é log, não regra** — `MATRIZ_INTERCOMPANY[acao]['cfop_esperado']` é informacional. Validar pós-PO confirmada apenas para log/audit.
- **Idempotência via dfe.state** — `buscar_dfe` reporta `status='processado'` se DFe já foi escriturado. Folha aborta antes do passo 2 e reporta IDEMPOTENT_DONE.
- **D17 ACAO_PARA_CFOP_ENTRADA** — ao registrar lotes localmente para auditoria, gravar CFOP 1xxx (não 5xxx — fiscal_position FB só tem cadastro para entrada).
- **G014 lote vencido** — sub-skill C5 `auditando-cadastro-fiscal-odoo` deve rodar pré-flight ANTES desta folha. Lote vencido na origem com saldo bloqueia picking auto-gerado.
- **Picking GERADO pelo Odoo (não criado por nós)** — o picking nasce da PO confirmada (passo 5). NUNCA criar picking de entrada manual aqui (isso é antipadrão AP2 — para caso DFe não veio, usar fluxo 1.2.2 que cria DFe manual, NÃO picking manual).
