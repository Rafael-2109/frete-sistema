# Fluxo 2.4 — Cancelar reserva órfã (cirurgia ou cancelamento)

**Quando:** após uma operação que zerou ou mudou o saldo de um quant (tipicamente `ajustando-quant-odoo --resetar-reserva`), restaram `stock.move.line` em `state=assigned` apontando para quants com `quantity=0` ou para saldos que mudaram. O picking continua em `assigned` mas o saldo subjacente não suporta a movimentação. Próxima tentativa de validar o picking gera estoque negativo ou erro.

**Galho da árvore (subagente):** `2 Estoque (sem NF) → 2.4 cancelar reserva / cirurgia em ML órfã / cancelar picking`.

---

## Premissas (pesquisar+validar ANTES de qualquer write)

1. **Identificar os quants alvo** (com MLs órfãs apontando para eles):
   - Tipicamente lista de `quant_id` que foi zerada por `ajustando-quant-odoo --resetar-reserva` em sessão anterior.
   - OU descobrir via `consultando-quant-odoo --incluir-qty-zero` filtrando quants com `quantity=0 AND reserved!=0`.
2. **Identificar as MLs órfãs** apontando para esses quants:
   - `stock.move.line` com `state IN [assigned, partially_available]`, `product_id+location_id+lot_id` matching os quants alvo.
   - Tipicamente 1 ML por (cod, lote, location) — checar.
3. **Identificar os pickings que contêm essas MLs**:
   - `picking.name` (ex.: `FB/OUT/01046`, `FB/INT/07950`).
   - **Estado do picking** (`assigned/confirmed/waiting`): se `done` → IMUTÁVEL, parar; se `cancel` → NOOP.
4. **Identificar o tipo de cirurgia POR PICKING**:
   - **Cirurgia** (`cancelar_moves_orfaos`): picking tem MLs OK além das órfãs → preservar picking, cancelar só órfãs.
   - **Cancelamento inteiro** (`cancelar_picking_inteiro`): todas (ou quase todas) MLs do picking são órfãs → cancelar via `action_cancel`.
5. **Verificar implicação fiscal** (cuidado especial para pickings de saída):
   - Se já houve NF emitida (`account.move` com `invoice_origin = picking.name`) → consultar Fiscal antes (pode exigir nota de crédito).
   - Se NF ainda não emitida → operação puramente técnica, seguir.
6. **Avaliar product_uom_qty do move parent** (para cirurgia):
   - Move com 1 ML órfã (1:1): `product_uom_qty=0` (zera demanda).
   - Move com várias MLs mistas (1 órfã + N OK): `product_uom_qty = sum(quantity das MLs OK)` (reduz demanda para o restante).

---

## Sequência (átomos componíveis; output de um alimenta input do próximo)

```
1) (HOST) Listar quants alvo + MLs órfãs + pickings afetados.
   Caminho 1: lista de quant_ids já conhecida (do log da sessão WRITE anterior).
   Caminho 2: descobrir via consultando-quant-odoo:
      python .claude/skills/consultando-quant-odoo/scripts/consultar_quants.py \
        --cods <cods> --empresas <emps> --incluir-qty-zero  # filtra qty=0
   → Pega quant_ids. Cross-com stock.move.line via search no Odoo.

2) (HOST) Para CADA picking afetado, decidir estratégia:
   - Se todas MLs são órfãs → cancelar inteiro (atomo 2).
   - Se mistura → cirurgia (atomo 1).
   - Calcular para cada move parent das MLs órfãs: novo product_uom_qty
     (0 ou soma das MLs OK do mesmo move).

3) (PARA CADA picking — cirurgia) operando-reservas-odoo:
   python .claude/skills/operando-reservas-odoo/scripts/operar_reserva.py \
     --picking-id <P> --ml-ids <ml1,ml2,...> \
     --moves-writes "m1:qty1,m2:qty2,..." --confirmar
   → output: {status=CIRURGIA_OK, picking_state_depois, moves_estado}

4) (PARA CADA picking — cancelamento inteiro) operando-reservas-odoo:
   python .claude/skills/operando-reservas-odoo/scripts/operar_reserva.py \
     --cancelar-picking --picking-id <P> --confirmar
   → output: {status=PICKING_CANCELADO, picking_state_depois='cancel'}

5) (HOST) ⚠️ OBRIGATÓRIO — zerar reserved_residual dos quants alvo:
   Após unlink das MLs, Odoo recalcula `reserved_quantity = reserved_atual - ml.quantity`.
   Se quants já estavam com reserved=0 (pós --resetar-reserva), ficam NEGATIVOS.
   Solução: stock.quant.write({reserved_quantity: 0}) via service.zerar_reserved_residual
   (atomo 3 da skill).

6) (HOST) consultando-quant-odoo → verificar estado final:
   python .claude/skills/consultando-quant-odoo/scripts/consultar_quants.py \
     --cods <cods> --empresas <emps> --excluir-indisp --agregar
   → esperado: quants alvo com qty=0 reserved=0; pickings em assigned (cirurgia)
     ou cancel (cancelamento).
```

---

## Gotchas do fluxo

- **`stock.move._action_cancel` é PRIVADO no XML-RPC** (Odoo 17 CIEL IT, validado 2026-05-23). Cancelar 1 move individualmente NÃO é possível via XML-RPC público. Workaround: `unlink` MLs + `write({'product_uom_qty': X})` no move parent. Codificado no átomo `cancelar_moves_orfaos`.
- **`reserved_uom_qty` NÃO existe** em Odoo 16/17 (G024). Usar `quantity` ou `qty_done` (legado).
- **Unlink ML causa reserved NEGATIVO** quando quant já está com reserved=0 — Odoo calcula `reserved_novo = reserved_atual − ml.quantity`. **OBRIGATÓRIO** chamar `zerar_reserved_residual` após cirurgia.
- **Picking em `done`** é IMUTÁVEL — átomos rejeitam (`FALHA_PICKING_STATE_INVALIDO`).
- **Picking em `cancel`** é NOOP — átomo cancelar_picking_inteiro retorna `NOOP` sem efeito.
- **MLs em `state='done'`** indicam consumo realizado — NÃO mexer (filtro `state IN [assigned, partially_available]`).
- **Cirurgia em multi-ML por move**: move com 2 MLs (1 órfã 0.40 + 1 OK 0.20) → ajustar `product_uom_qty` para 0.20 (soma das MLs OK que sobram). NUNCA zerar se sobrar ML OK no mesmo move.
- **Implicação fiscal**: pickings de saída (FB/OUT/*) podem ter NF emitida. Verificar `account.move.invoice_origin = picking.name` ANTES.

## Exemplo (caso real 2026-05-23 — 6 pickings)

```bash

SK9=.claude/skills/consultando-quant-odoo/scripts/consultar_quants.py

# 1) Cirurgia: FB/FB/EMB/11673 (1 ML órfã, mas move parent tem 2 MLs — 1 órfã 0.40 + 1 OK 0.20)
python "$SK24" --picking-id 316701 \
  --ml-ids 217654353 \
  --moves-writes "1075205:0.1986" \
  --confirmar
# → CIRURGIA_OK

# 2) Cancelamento inteiro: FB/INT/07950 (única ML é órfã)
python "$SK24" --cancelar-picking --picking-id 320076 --confirmar
# → PICKING_CANCELADO (state→cancel)

# 3) OBRIGATÓRIO após cirurgias: zerar reserved residual
# (chamada via service inline, ainda sem CLI dedicado — implementar quando aparecer caso)
# svc.zerar_reserved_residual([258975, 258988, ..., 261592])

# 4) Verificar
python "$SK9" --cods 4856125,105000016,104000037 --empresas FB --excluir-indisp --agregar
```
