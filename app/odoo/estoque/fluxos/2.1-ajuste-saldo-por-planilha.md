# Fluxo 2.1 — Ajuste de saldo por planilha (Família A)

**Quando:** o usuário envia 1 XLSX com N linhas `(EMP|filial, COD|cod, LOTE|lote, AJUSTE|diff_qtd|qtd|AJUSTE POSITIVO)` para corrigir o saldo de N quants no Odoo (resíduo, contagem, criar saldo). Cada linha = 1 ajuste atômico de 1 quant (sem mover saldo entre origens/destinos — para isso ver fluxo 2.2).

**Galho da árvore (subagente):** `2 Estoque (sem NF) → 2.1 ajuste de saldo`. Caso "1 quant pontual" usa direto a skill `ajustando-quant-odoo` (sem precisar dessa folha). Caso "N quants via planilha" = ESTA folha.

---

## Premissas (pesquisar+validar ANTES de qualquer write)

1. **Planilha:** arquivo XLSX (formato livre, schema detectável). Colunas obrigatórias normalizadas:
   - `empresa`/`filial`/`EMP` ∈ {`FB`, `CD`, `LF`} → mapeia para `company_id` via `app/odoo/estoque/_utils.py:EMPRESAS`/`resolver_empresa` (FB=1/8, CD=4/32, LF=5/42).
   - `cod`/`COD` = `default_code` do `product.product`. **1 ativo esperado** — se 0 ou >1, parar e perguntar.
   - `lote`/`LOTE` = `name` do `stock.lot`. Pode vir vazio se `product.tracking == 'none'` (resolver via `_utils.resolver_produto`).
   - `delta`/`AJUSTE`/`diff_qtd`/`qtd`/`AJUSTE POSITIVO` = float (6 casas). **Decidir a semântica do sinal pelo cabeçalho**:
     - `AJUSTE` (script 11/12): cabeçalho diz "negativo" ou "positivo" — usar só linhas do sinal correto.
     - `diff_qtd` (script 13): + cria/aumenta, − reduz.
     - `qtd`/`AJUSTE POSITIVO` (14_v2/criar_saldo): sempre +.
2. **Modo da skill por linha** (decidir 1 vez, aplicar a todas):
   - `--delta <X>` (somar/subtrair) — caso comum (planilhas dos scripts 11/12/13/14_v2/criar_saldo_lf).
   - `--criar-se-faltar` quando `Δ > 0` E o lote pode não existir ainda (12/13/14_v2/criar_saldo_lf). NÃO usa com `--delta < 0`.
   - `--valor-absoluto` apenas se a planilha trouxer "saldo definitivo" (não é o caso atual).
3. **Empresas únicas:** 1 planilha ≠ 1 empresa. Filtrar linhas por `empresa` antes de processar — cada chamada à skill recebe 1 empresa via `--empresa`.
4. **Saldo vivo:** entre dry-run e `--confirmar` o saldo pode mudar (operação viva). Rodar o `--dry-run` **logo antes** do `--confirmar` da mesma linha.
5. **Reservas/MOs prendendo o quant:** se `qty_apos < reservada`, a skill bloqueia (`FALHA_RESERVADO`). Caso real: liberar a reserva ANTES (fluxo 2.4 — `operando-reservas-odoo`, em construção) — não compor automaticamente nesta folha.

---

## Sequência (átomos componíveis; output de um alimenta input do próximo)

```
1) (HOST/orquestrador) carregar planilha → lista[ {empresa, cod, lote, delta, modo} ]
   - pandas.read_excel(...) com dtype={cod:str, lote:str}
   - normalizar headers (case-insensitive), validar colunas obrigatórias
   - filtrar/agrupar por empresa (1 lote por execução opcional)
   - ground-truth: padrão dos scripts-fonte (11/12/13/14_v2/criar_saldo_positivo_lf)

2) (PARA CADA linha) ajustando-quant-odoo --dry-run
     args (modo chave):
        --cod <COD>
        --empresa <FB|CD|LF>
        --lote <LOTE>          # omitir se tracking=none
        --delta <Δ>             # ou --valor-absoluto <V>
        [--criar-se-faltar]     # se Δ>0 e lote/quant pode não existir
        [--resetar-reserva]     # só se houver reserva órfã/negativa conhecida
     → output: {modo:'dry-run', chave:{...}, resultado:{status, qty_antes, qty_apos, ajuste_aplicado, ...}}
   - Status terminais OK: DRY_RUN_OK · NOOP
   - Falhas que param a linha (não a planilha inteira): FALHA_PRODUTO, FALHA_LOTE, BLOQUEADO_SERIAL, FALHA_QUANT_NEGATIVO, FALHA_QUANT_VAZIO, FALHA_RESERVADO, FALHA_ODOO

3) (HOST) consolidar plano:
   - Tabela: idx | cod | lote | empresa | Δ | qty_antes | qty_apos | status
   - Apresentar ao usuário (NÃO confirmar sem OK explícito).
   - Soma absoluta dos ajustes (sanity-check vs planilha).

4) (PARA CADA linha aprovada) ajustando-quant-odoo --confirmar
   - Mesmos args do passo 2 + `--confirmar`.
   - Verificar status: EXECUTADO (esperado), NOOP (Δ resultou em 0 e quant já estava lá).
   - Em falha durante a confirmação: logar a linha, seguir para a próxima (planilha não é atômica).

5) (HOST) Log JSON estruturado (replicar formato dos scripts-fonte):
   - Salvar em `scripts/inventario_2026_05/auditoria/log_2.1_ajuste_planilha_<ts>.json`
   - Campos: args, total, contagem_status, contagem_lote_acao, contagem_quant_acao, soma_ajustada, resultados[].
```

## Gotchas do fluxo

- **G028 (consolidar_move_lines)** — afeta picking, não esse fluxo (ajuste puro). Se a planilha vier de uma reconciliação pós-picking, conferir reservas antes (passo 5 da Premissa).
- **action_apply_inventory infla quant NEGATIVO** (memória `gotcha_inventory_adjustment_quant_negativo`) — a skill protege via `validar_nao_negativar` e bloqueia (`FALHA_QUANT_NEGATIVO`). Para destino negativo, usar fluxo 2.5 (`operando-picking-odoo`).
- **`=` em `stock.lot.name` retorna vazio intermitente** (G002) — `StockLotService.buscar_por_nome` já usa `in [valor]` + fallback `=like`. Não duplicar essa lógica na planilha.
- **Lote multi-empresa** (memória `gotcha_lote_multiempresa_company_filter`) — `criar_se_nao_existe(name, pid, company_id)` SEMPRE filtra por `company_id`. Mesmo lote físico pode existir em FB+LF; sem filtro → "Empresas incompatíveis".
- **2 lotes MIGRAÇÃO/produto** — quando o produto tem MIGRAÇÃO/MIGRACAO/MIGRACÃO etc., a busca por nome pode achar múltiplos. Confirmar pelo `id` no dry-run antes de `--confirmar`.
- **`--criar-se-faltar` é silenciosamente ignorado se o lote já existe** (skill faz `reused`) — comportamento correto, mas operadores às vezes esperam ver `created`. Mensagem clara no resumo.
- **6 casas decimais** — `round(Δ, 6)`. Se a planilha vier com `e-08`, anotar `ajuste_truncado_em_6_casas=true` (não é falha).
- **Worktree sem `.env`** — para rodar dry-run/confirmar local, carregar só `ODOO_*` da árvore original: `set -a; . <(grep -E '^ODOO_' /home/.../frete_sistema/.env); set +a` (não exportar `DATABASE_URL` — evita tocar PROD do banco local).

## Exemplo (caso real, validado em 2026-05-23)

Planilha `/mnt/c/.../Downloads/AJUSTE SALDO CD.xlsx` (35 linhas, EMP=CD, AJUSTE>0). Linha 1: `COD=205460830  LOTE=345232-25  AJUSTE=+0.00002`.

```bash
# Skill — passo 2 (dry-run; default; exit 4 se OK)
SK=.claude/skills/ajustando-quant-odoo/scripts/ajustar_quant.py
python "$SK" --cod 205460830 --empresa CD --lote 345232-25 --delta 0.00002
# → status=DRY_RUN_OK  product_id=28010  lot_id=56738  quant_id=219019  qty 682.69 -> 682.69002

# Skill — passo 4 (após OK do usuário; exit 0 se EXECUTADO)
python "$SK" --cod 205460830 --empresa CD --lote 345232-25 --delta 0.00002 --confirmar
```

Validação C6 (3 casos cruzados contra `frete_sistema/scripts/inventario_2026_05/auditoria/log_*.json`):
- script 11 (ajuste negativo CD) · cod=4310152 lote=119338 Δ=−1.1e-05 → premissas 100% match.
- script 12 (ajuste positivo CD lote existente) · cod=205460830 lote=345232-25 Δ=+2e-05 → 100% match.
- criar_saldo_positivo_lf · cod=104000127 lote=`0730/682153-F3` LF Δ=+37.5 `--criar-se-faltar` → premissas 100% match.

> Operação viva: `qty_antes`/`qty_apos`/`status` divergem do ground-truth quando o saldo mudou após o run original — comportamento esperado, **não é falha** do átomo.
